# runtime-business-spec Skill 设计方案

> 定位：认知管线的**业务语义层 / 终端业务产物**。消费 ui-collab 已确认的界面事实 + flow-map 代码锚，
> 深探 Service 层抽业务规则，**串成端到端流程并翻成纯业务语言**，产出一份业务人员可读的
> 「运行时业务流程与规则说明」。
>
> 上游：runtime-capture → runtime-flow-mapper → runtime-ui-collab → **runtime-business-spec（本 skill）**
> 本 skill **替代** runtime-spec-mapper（其 L0-L2 规约无真实下游消费者，设计另有问题，退役）。

---

## 〇、设计立场（为什么是这个位置、这个职责）

1. **管线的终端业务产物**。上游层层收敛：capture（原始证据）→ flow-map（落锚骨架）→ ui-collab（人工确认界面事实）
   → 本 skill（翻成业务语言 + 串流程 + 补规则）。读者是**业务人员**，不是工程。
2. **接管 spec-mapper 的深探，唯一规则源**。spec-mapper 退役后，"下 Service 抽业务规则"只在本 skill 做一次——
   **无双事实源、无重复深探**。
3. **与 ui-collab 互补不重叠**：ui-collab 止于界面事实层（故意不串流程、不下 Service 抽规则）；
   本 skill 正好补这两件——**跨界面串流程** + **深探业务规则**。
4. **借 L0-L2 的内核纪律，弃其外形**：借 INV（规则=精确约束）/ 业务实体 / 状态机 / 现状≠规范 / 业务语言纯净；
   **不借** 用户故事结构、技术编号（INV-n/AC-n）、溯源锚（类.方法）、L0/L1/L2 层级标签——这些会把它退化成"换皮 spec"。

---

## 一、触发与输入

**触发**：「梳理运行时业务流程与规则」「出一份业务可读的现状流程说明」「把界面文档串成业务流程」。

**用户提供**：
- `{name}`（定位上游产物）
- 前端工程目录 + 后端工程目录；CodeGraph + GitNexus 索引（深探用）

**输入三层（全复用，不新增采集）**：

| 层 | 来源 | 用途 |
|----|------|------|
| **界面事实（已确认）** | `docs/prd/{name}/*.md`（ui-collab，**status=confirmed**） | 界面业务事实 + `nav_context` 串流程 + 五维上下文 |
| **流程骨架 + 代码锚** | `docs/runtime-flow/{name}/{name}_flow-map.md` | 端到端结构 + 定位 Service 入口 |
| **规则源** | 工程代码 + 索引 | 深探 Service 层抽校验/计算/状态规则 |

> **不直读 capture 原始证据**——界面事实 ui-collab 已确认，本层消费"已确认产物"，逐层收敛。
> ui-collab 中 `skipped`/未梳理的界面，在流程中标「待核实·未梳理」，不假装完整。

---

## 二、不变量（核心纪律）

0. **前置闸门（强制·首步）**：查 flow-map + ui-collab **confirmed** 文档 + 工程/索引就位；缺则**中断 + 提示先跑上游**，绝不臆造。
1. **零技术术语**：正文无接口路径、代码路径、类名、字段名、标注黑话（`[码]`/`~inferred` 等）——**纯业务语言**。
2. **规则源自深探、确信度业务化**：每条规则从 Service/校验逻辑深探而来（不造规则），按 INV 纪律写成精确约束，
   带业务化确信度词 **〔已核实〕/〔待核实〕**。
3. **串流程依 `nav_context`**：跨界面流程严格按 ui-collab 衔接契约串，不臆造跳转。
4. **冲突优先级**：界面/操作事实以 ui-collab **人工确认为最高**；规则细节以**代码深探**为准；二者冲突时人工确认优先。
5. **现状 ≠ 规范**：只述系统「当前这样跑」，不评判「应该怎样」；疑似防御兜底/遗留逻辑标〔待核实〕，不当业务规则断言。
6. **深探有界**：到 Service + 校验 + 状态机即止，**不下 DAO/SQL**。
7. **不越界 / 禁杜撰**：不采集、不落锚、不改上游产物；只翻译 + 串流程 + 补规则。

## 三、工序（强制顺序，每阶段一个 Checkpoint）

```
⓪ 前置闸门  →  ① 串流程（端到端）  →  ② 深探提规则  →  ③ 翻译产出  →  自校验
```

### ⓪ 前置闸门
查 flow-map + ui-collab confirmed 文档 + 工程/索引。齐全 → ①；缺则中断提示。
- **Checkpoint**：confirmed 界面文档存在；skipped/缺失界面已列为待核实项。

### ① 串流程（端到端）
按各 ui-collab 文档 `nav_context` 串成端到端流程，识别「角色 + 环节」时序，画**纯业务节点** Mermaid（无接口/无路径）。
- **Checkpoint**：流程闭环；节点为业务动作（非技术事件）；未梳理界面已显式标注。

### ② 深探提规则（核心增值）
沿 flow-map 代码锚深探 Service：`callees → Service 主逻辑 → DTO/VO（业务实体+属性）→ 枚举/常量（状态机）→ 前端校验`。
抽每个环节的业务规则（校验/计算/状态转换），逐条按 INV 纪律落成精确约束。**到 Service 止，不下 DAO。**
- **Checkpoint**：每环节规则可回指代码（内部记忆，不写进正文）；代码确证 vs 单样本/未覆盖分别归〔已核实〕/〔待核实〕。

### ③ 翻译产出
技术证据 → 纯业务语言，套用产物结构写入 `docs/business-flow/{name}/{name}_业务流程说明.md`。
- **Checkpoint**：正文零技术术语；每条规则带确信度词；状态流转/角色权限齐。

### 自校验
- 零术语扫描：正文无 `/api/`、`.java`、`.vue`、类名、`[码]`/`~inferred` 等。
- 完整性：每环节有规则 + 产出数据；未梳理项已标〔待核实·未梳理〕。

## 五、确信度词（业务化，替代技术标注）

| 词 | 含义 | 来源 |
|----|------|------|
| 〔已核实〕 | 代码确证 / 人工已确认，可信 | ui-collab `[人工]`/`[runtime+码]`、代码深探确证 |
| 〔待核实〕 | 单样本观察 / 推断 / 未覆盖分支 / 未梳理界面，待人核 | ui-collab `~待确认`、深探未覆盖、skipped 界面 |

> 两档即可。深溯源不内置——需回溯时结合本文档 `source_inputs`（ui-collab 文档 + flow-map）查。

## 六、与上下游 Skill 的关系

```
runtime-capture     ← 采集运行时操作与 API 时序
    ↓
runtime-flow-mapper ← 落锚骨架（flow-map：界面-接口-代码）
    ↓
runtime-ui-collab   ← 人工确认界面事实（不串流程、不抽规则——故意留给本 skill）
    ↓
runtime-business-spec ← 【本 skill】串流程 + 深探规则 + 翻成纯业务语言（替代 spec-mapper）
```

> 借 L0-L2 内核（INV/实体/状态机/现状≠规范/语言纯净/术语表），弃其外形（用户故事轴/技术编号/溯源锚/层级标签）；
> 守三道闸不退化为规约：**组织轴=流程环节 · 读者=业务 · 零术语+确信度词**。
