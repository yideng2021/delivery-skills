# runtime-ui-collab Skill 设计方案（重构版）

> 定位：管线**首个人工闸口**。以 runtime-flow-mapper 的 flow-map 为骨架、runtime-capture 的原始证据
> （session_log / records.ts / api_details）为界面操作事实源，通过人机协同逐界面核对，
> 以「runtime 原始证据 + 浅层代码回查」为依据，把推断（`~inferred`）收敛为人工确认（`[人工]`/`[runtime+码]`），
> 输出高质量、已校准的**界面业务事实文档集**。
>
> 上游：runtime-capture → runtime-flow-mapper → **runtime-ui-collab（本 skill）**
> 下游（被动消费本产物，本 skill 不感知）：runtime-spec-mapper（业务规约）、流程总览 skill（串界面成流程）

---

## 〇、设计立场（为什么是这个位置、这个职责）

1. **本 skill 是管线第一个有人在场的环节**。上游 flow-mapper / spec-mapper 全是 AI 自动产出，带推断债。
   本 skill 专门把界面级 `~inferred` 校准成 `[人工]`，产出**被下游当作权威依据消费**的已确认界面事实。
2. **读 capture 原始证据，但彻底不读 spec**。
   - **要读 capture 原始产物**：flow-map 是 flow-mapper 落锚后的蒸馏件，它把 ui-collab 最需要的"界面操作粒度"
     （真实触发方式、界面内操作序列、录到的字段）**摘要压没了**。这些 ground truth 在 `records.ts`/`session_log`/`api_details` 里。
     ui-collab 是全管线**唯一以"界面/操作粒度"本身为产出**的 skill，故必须直接消费这批原始证据，否则只能回头猜/问人。
   - **不读 spec**：spec-mapper 在本 skill **之后**，读它会反向继承其推断债，违背"先校准事实、再做业务抽象"的目的。
3. **职责边界：只做"界面事实层浅回查确认"**。
   - 本 skill 确认「**界面上发生了什么**」——功能点做什么、怎么触发、有哪些字段/操作。
   - 业务规则 INV / 实体抽象 / 完整状态机 是**下游 spec-mapper 的深探职责**，本 skill 不碰，避免重叠。
   - 一句话：**消费 runtime 证据 + 浅回查锚点验证，不下 Service 抽业务规则。**
   - 与 flow-mapper **不重叠**：flow-mapper 用这批产物做"落锚到代码"，本 skill 用同一批做"界面操作梳理"，
     同源证据、不同目的，属产物级解耦的正常消费。

---

## 一、触发与输入

**触发**：「逐界面梳理运行时业务」「把 flow-map 整理成界面文档」「界面业务核对」「人机协同界面梳理」。

**用户提供**：
- `flow-map` 文件路径（必须）
- capture 工作目录 `{工作目录}/{name}/`（必须，取原始证据）
- 前端工程目录 + 后端工程目录（必须，代码回查用）
- CodeGraph + GitNexus 索引（前后端两仓，代码回查用）

**输入三层（各司其职）**：

| 层 | 来源 | 在本 skill 的用途 |
|----|------|------------------|
| **骨架/索引** | `flow-map.md` | 节点→界面映射、代码锚、流程结构、三证表 |
| **界面操作事实源** | `session_log` | 导航序列(前置/后置界面) + 每界面 route(识别主键) + 界面内 API 时序 |
| | `records.ts` | 真实触发方式 + 界面内操作序列 + 录到的输入字段 → 喂功能地图&界面流程图 |
| | `api_details` | 实际 req/resp 字段 payload（**按需**加载）→ 功能点字段/数据 |
| **验证/补全** | 工程代码 + 索引 | 浅回查补全"这个功能完整能做什么"（分支/字段） |

> `api_requests` flow-map 已消化，一般不必直读；HAR 不消费。
> **不消费 spec**（理由见〇.2）。

## 二、不变量（核心纪律）

0. **前置闸门（强制·首步）**：校验 flow-map.md 存在/合法、capture 工作目录原始产物就位、工程/索引就位；
   缺则**中断 + 提示先跑上游**，绝不臆造。
1. **推断必收敛、缺证不断言**：
   - `~inferred`（AI 推断）必须经"runtime 证据/回查码自证"或"问人确认"收敛，**confirmed 文档残留=0**。
   - **代码找不到的逻辑不得直接断言**：回查无源码佐证的行为/规则，标 `~待确认`，经协同仍未定论则**允许显式残留**（计入 confidence，不算 inferred 残留）。
2. **runtime 证据优先 → 回查补全 → 问人兜底**（三级，不可越级）：
   - 先查 runtime 原始证据（`records.ts`/`session_log`/`api_details`——"实际发生了什么"的直接证据）；
   - 再代码回查补全（"完整能做什么"）；
   - 仅**业务判断类**（给谁用 / 业务意图 / 边界规则）才留给人。
   > 反例：「触发方式=点编辑按钮」不该当推断问用户——records.ts 录到了真实点击目标，直接读出标 `[runtime+码]`。
3. **跨服务调用以 runtime 为存在性权威**：代码回查 **0 命中 ≠ 无此逻辑**。session_log/api_requests 已证调用真实发生 →
   标 `[runtime]` + 注"代码不在本工程（疑外部/微前端/共享组件）"，**禁止下'无此逻辑'结论**；不与 flow-map 已有 `[外部:xxx]`/`~unresolved` 裁决冲突。
4. **时序 ≠ 因果**：界面内相邻接口不等于因果绑定。**要断言"因果/存在中间步骤"时**，须用 api_details 请求体字段差异佐证
   （同名接口多次调用的字段变化）；佐证不足则**只记时序、不标因果**。非断言因果时不强制逐条 diff。
5. **标注对齐上游 + 增 `[人工]`/`~待确认`**：沿用上游 `[码]`/`[runtime]`/`~inferred`/`~unresolved`；
   新增 `[人工]`（问人已定）与 `~待确认`（问人未定）；人工确认后升格 `~inferred` → `[人工]`/`[runtime+码]`，原标注移除。
6. **即时落盘 + 状态机读**：每界面草稿即写 md（`status: draft`），确认升 `confirmed`、跳过置 `skipped`；
   重启按 frontmatter `status` 断点续做，任意中断不丢进度。
7. **不越界 / 禁杜撰**：不重做上游（不采集 / 不落锚），不做下游（不抽业务规则、不串跨界面流程），
   只产已确认的界面事实文档。

## 三、工序（强制顺序）

```
⓪ 前置闸门  →  ① 界面范围确认  →  ②~N 逐界面人机协同（每界面即时落盘）  →  收尾汇总
```

### ⓪ 前置闸门（Preflight）

校验 flow-map.md 存在/合法、前后端工程与索引就位。齐全 → 进入 ①；缺 flow-map → 中断提示。

---

### ① 界面范围确认

1. AI 读 flow-map 节点清单 + 三证表，**以 session_log 的 route/URL 为主键**推断「节点 → 界面」映射
   （节点↔界面是 N:M：一节点可能是子步骤、多节点可能是同界面不同状态）。
2. 展示界面清单表格，用户确认或修正。
3. 确认后进入逐界面协同。

| 界面ID | 界面名称 | route | 类型 | 对应 flow-map 节点 | 触发入口 |
|--------|---------|-------|------|-------------------|---------|
| UI-01 | ... | /xxx/list | 独立/合并/子界面/多视角 | 节点N | ... |

---

### ②~N 逐界面人机协同（每个界面循环，确认后即时落盘）

**Step 1 — 草稿生成**

AI 基于 **flow-map 骨架（限定本界面节点/route）+ runtime 原始证据（records.ts/session_log/api_details 的本界面切片）+ 浅层代码回查**生成当前界面草稿：
- 业务介绍（是什么 / 谁用 / 能做什么 / 关键约束）
- 上下文说明（前置 / 后置 / 数据 / 权限 / 状态 五维度，前置/后置取自 session_log 导航序列）
- 功能地图（骨架树形 + 每功能点描述，**按界面操作/功能点组织**——触发方式/操作序列/字段取自 records.ts，区别于 spec 的角色故事切法）
- 界面流程图（Mermaid，结构取自界面内操作时序）

> **按界面切片读取**：flow-map 为索引定位本界面对应节点/route，再据此只读 records.ts/session_log/api_details 中
> 本界面的相关片段（api_details 按需），不整批灌入，契合 capture 两段式加载。

草稿中所有未确证内容标 `~inferred` / `~unresolved`。

**Step 2 — 推断收敛（runtime 证据优先 → 回查补全 → 问人兜底）**

AI 自动识别草稿中的 `~inferred` / `~unresolved` 点，按三级处理：
1. **runtime 原始证据自证**：先查 records.ts/session_log/api_details——多数"触发方式/操作序列/字段"在此直接命中，标 `[runtime+码]`/`[runtime]`。
   - 跨服务接口代码 0 命中时，**以 runtime 为存在性权威**标 `[runtime]`+注"代码不在本工程"，不判"无此逻辑"（不变量3）。
   - 推因果/中间步骤前，先 diff 同名接口请求体字段变化佐证；佐证不足只记时序（不变量4）。
2. **代码回查补全**：runtime 未覆盖的，用 flow-map 锚点浅回查代码自证（到锚点验证为止），升格标注。
   **无源码佐证的逻辑标 `~待确认`，不直接断言**（不变量1）。
3. **问人兜底**：仅业务判断类（给谁用 / 业务意图 / 业务边界）回查无法定论的，才向用户追问；
   问人已定 → `[人工]`，问人未定（选"保留标注"）→ `~待确认` 显式残留。

追问呈现规则：
- **复杂推断**（业务意图、边界规则、runtime+代码均未定论）→ 卡片式

```
┌─ ❓ 待确认 ──────────────────────────────────────────┐
│ 问题：[功能X] 面向哪类角色 / 业务边界                │
│ 我的推断：仅审批人可见此操作                         │
│ 推断依据：records.ts 单样本 + 代码无显式权限校验     │
│ 推荐答案：是                                         │
│ 可选：[是] [否，实际是______] [不确定，保留标注]     │
└──────────────────────────────────────────────────────┘
```

- **简单确认**（字段名、操作文案）→ 对话式直接问

**Step 3 — 用户答 → 升格标注 → 即时写盘 → 确认菜单**

AI 据回答更新草稿与标注（`~inferred` → `[人工]`），**立即写入该界面 md（`status: draft`）**，再展示确认菜单。
选 `[1]` 过升档自检后置 `status: confirmed` + `confirmed_at`；选 `[5]` 置 `status: skipped`：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  UI-01 · [界面名称] — 草稿已生成 / 已落盘
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  [1] 全部确认，进入下一界面
  [2] 修正业务介绍 / 上下文
  [3] 修正功能地图
  [4] 补充遗漏功能
  [5] 跳过此界面
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> 即时落盘替代"收尾统一输出"：每界面确认即写盘，收尾只做汇总，任意中断不丢进度。
> **断点续做**：重启时扫 `docs/prd/{name}/` 各文件 frontmatter `status`，`confirmed`/`skipped` 跳过、`draft` 续做。

---

### 收尾阶段

所有界面处理完毕后汇总：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  所有界面协同完成（均已落盘，按 status 统计）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  confirmed：UI-01 / UI-02
  skipped  ：UI-03
  draft 残留：无
  产物目录：docs/prd/{name}/
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 四、输出

**目录**：AI 扫描工程现有 `docs/prd/` 结构，推断输出路径，确认后写入。

**文件命名**：`{界面名称}.md`（每界面一文件）。

**文档结构**（每个界面文件，文件头借鉴 design-writer 的 frontmatter 模式，**选择性裁剪**）：

```
---
doc_type: ui-business-map                 # 固定标识，供下游机读识别
ui_id: UI-01                              # 稳定身份（保护文档名：下游按 ui_id 引用，不依赖文件名）
ui_name: {界面名称}
route: /xxx/list
status: draft                             # draft（草稿待确认）| confirmed（人工确认）| skipped（跳过）
source_refs:                              # 引用保护(SSOT)：记来源锚，不复述原文
  flow_map: docs/runtime-flow/{name}/{name}_flow-map.md#节点4
  capture:  {工作目录}/{name}/             # session_log / records.ts / api_details
  code:     [src/views/.../Index.vue, com.xxx.XxxController]
upstream_nodes: [节点4, 节点5]             # 对应 flow-map 节点
nav_context: {prev: UI-00, next: UI-02}   # 衔接契约：前置/后置界面（下游串流程用）
confidence: {runtime+码: 8, 人工: 3, 待确认: 1, inferred残留: 0}   # 收敛自检；inferred残留=0 为升档硬指标，待确认可残留
generated:    {ISO}
confirmed_at: {ISO | —}
---
> 现状(as-is)界面业务事实 · 非规范性 · 仅供参考

# {界面名称} — 界面业务梳理
## 一、界面业务描述
  1.1 业务介绍
  1.2 截图（占位 —— capture 无截图产物，人工后补）
  1.3 上下文说明（前置/后置/数据/权限/状态 五维）
## 二、界面功能地图
  2.1 功能骨架总览
  2.2 功能详细描述
## 三、界面流程图
```

**借鉴 design-writer 的三点（其余规约链字段/I8–I11/CDR 不引入——ui-collab 是参考件不进 validate 链）**：

- **frontmatter 文件头**：把原"衔接契约头"上提为结构化 frontmatter，下游（spec-mapper / 流程总览）**机读**而非解析正文。
- **`status` 状态机**：`draft → confirmed / skipped`，落实即时落盘 + 断点续做——恢复时扫各文件 `status` 即知进度，收尾汇总直接读 `status` 统计。
- **引用保护(SSOT) + 顶部非规范声明**：`source_refs` 只记来源锚不复述上游原文；顶部声明对齐 spec-mapper 的"现状·非规范·仅供参考"。

> 本 skill **不做**跨界面流程串联（留给下游"流程总览 skill"），但 `nav_context` 留好契约让它能把界面集重新串成流程。

**status 升档自检**（借鉴 design-writer §11；`draft → confirmed` 前必过）：
`~inferred 残留 == 0`（`~待确认` 允许残留，但须显式列入 `confidence`）、五维上下文齐、功能地图无空项、
`nav_context`/`source_refs` 填全——任一不满足不得升 `confirmed`。

## 五、确信度标注规范（对齐上游）

| 标注 | 含义 |
|------|------|
| `[runtime+码]` | 运行时录制 + 代码双重验证（最高） |
| `[码]` | 代码直接验证 |
| `[runtime]` | 仅运行时见到，代码未深入（**含跨服务 0 命中**：调用真实发生但代码不在本工程） |
| `[人工]` | **本 skill 新增**：问人**已定**（由 `~inferred` 升格而来） |
| `~待确认` | **本 skill 新增**：代码无源码佐证、问人**未定**的逻辑断言；**允许显式残留**，计入 `confidence`，与 `[人工]` 配对 |
| `~inferred` | AI 推断，未直接验证（收敛前临时态，confirmed 产物中**残留=0**） |
| `~unresolved` | 本服务前缀但代码未找到 |

## 六、与上下游 Skill 的关系

```
runtime-capture        ← 采集运行时操作与 API 时序（session_log/records.ts/api_details…）
    │   ↓（落锚蒸馏）        ╲
runtime-flow-mapper        ╲（原始证据直供本 skill 取界面操作粒度）
    │  flow-map：界面-接口-代码三证（骨架/索引）  ╲
    ↓                                              ↓
runtime-ui-collab      ← 【本 skill】flow-map 骨架 + capture 原始证据 + 浅回查，
    │                     人机协同确认，输出已校准界面事实文档集
    ↓
    ├─→ runtime-spec-mapper  ← 下游：以已确认界面事实+代码深探出业务规约（待重构）
    └─→ 流程总览 skill        ← 下游：按衔接契约把界面集串成端到端流程（未来）
```

> 本 skill 同时消费 flow-map（骨架）与 capture 原始证据（界面操作事实）；与 flow-mapper 同源不同目的，零耦合。

> 与下游**产物级零耦合**：本 skill 不感知下游，只产已确认界面事实 + 衔接契约头；
> 下游按契约被动消费。
