# Baseline Init — 基线初始化与架构治理

> **触发条件**：当用户要求对一个现有项目进行"项目文档初始化"、"架构治理"、"基线梳理"时，自动激活此 Skill。

---

## 0. 元约束 (Meta-Constraints)

| #   | 约束                         | 说明                                                                                           |
| --- | ---------------------------- | ---------------------------------------------------------------------------------------------- |
| M1  | **契约生成，不修改业务代码** | 严禁在本 Skill 流程中生成或修改任何业务代码文件。所有产出仅限 `docs/` 目录下的 Markdown 文档。 |
| M2  | **注释驱动精炼 (CDR)**       | 通过 HTML 注释 `<!-- REVIEW: ... -->` 接收内联修改意见，更新后自动移除注释。                   |
| M3  | **逆向推导**                 | 当缺失需求或设计文档时，必须基于代码结构逆向推导，并在文档中标注 `[INFERRED]`。                |
| M4  | **幂等可重入**               | 每个阶段可独立重跑而不破坏其他阶段的产出，已存在的文件以增量方式更新。                         |
| M5  | **模板驱动**                 | 所有产出必须基于 `templates/` 中的模板生成，保证格式一致性。                                   |

---

## 1. 核心工作流 (Core Workflow)

> ⚠️ **强制顺序执行** — 必须按阶段编号顺序推进，前一阶段未完成不得进入下一阶段。

### 阶段一：工程锚定 (Anchoring)

**目标**：建立对项目的全面认知，确立开发规范"宪法"。

#### Step 1.1 — 工程初始化分析

| 项目     | 内容                                                                                                                               |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **动作** | 扫描项目根目录，识别编程语言、框架版本、构建工具、现有代码模块划分、已有文档资产。                                                 |
| **产出** | `docs/analysis/agent.md`                                                                                                           |
| **模板** | [`templates/agent.md`](templates/agent.md)                                                                                         |
| **要求** | ① 工程画像（语言/框架/构建工具/当前状态）<br>② 目录结构扫描（识别关键模块）<br>③ 技术栈依赖分析（类别/名称/版本/用途）<br>④ 初始化路径建议 |

**验收标准**：
- [ ] 包含完整的项目画像（名称、语言、框架版本、构建工具、状态）
- [ ] 目录结构扫描覆盖 ≥ 90% 一级目录
- [ ] 依赖分析包含核心框架、数据库驱动、中间件三大类别

#### Step 1.2 — 建立立法规范

| 项目     | 内容                                                                                                                                              |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| **动作** | 基于工程现状，确立覆盖 API、数据库、工程结构的开发规范。                                                                                          |
| **产出** | `docs/standards/api_rules.md`<br>`docs/standards/db_rules.md`<br>`docs/standards/project_rules.md`<br>`docs/standards/testing_rules.md` |
| **模板** | [`templates/api_rules.md`](templates/api_rules.md)<br>[`templates/db_rules.md`](templates/db_rules.md)<br>[`templates/project_rules.md`](templates/project_rules.md)<br>[`templates/testing_rules.md`](templates/testing_rules.md) |
| **要求** | ① 接口规范（RESTful/RPC 风格、URL 前缀、响应格式、错误码）<br>② 数据库命名与索引规范<br>③ 工程目录结构规范<br>④ 统一错误码体系<br>⑤ Git 提交规范<br>⑥ 单元测试与 TDD 测试规范（Mock 策略、框架选型、命名规范等） |

**验收标准**：
- [ ] API 规范包含协议、请求规范、响应结构（含分页与错误）
- [ ] 数据库规范包含命名规范、基础配置、通用字段定义、索引策略
- [ ] 工程规范包含目录结构、错误码定义、Git 提交规范、单元/集成测试与TDD规范

### 阶段二：核心设计 (Core Design)

**目标**：将业务需求转化为系统级技术蓝图。

#### Step 2.1 — 梳理总体需求规约

| 项目     | 内容                                                                                                                                  |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **动作** | 结合历史文档与代码结构推导，拆解系统边界与业务背景。                                                                                  |
| **产出** | `docs/design/requirements.md`                                                                                                         |
| **模板** | [`templates/requirements.md`](templates/requirements.md)                                                                              |
| **要求** | ① 业务背景与目标<br>② 核心业务流程（Mermaid 流程图）<br>③ 功能模块清单（模块/功能点/优先级/描述）<br>④ 非功能性需求（性能/安全/可用性 SLA） |

#### Step 2.2 — 构建总体系统架构

| 项目     | 内容                                                                                                              |
| -------- | ----------------------------------------------------------------------------------------------------------------- |
| **动作** | 将需求转化为技术蓝图。                                                                                            |
| **产出** | `docs/design/architecture.md`                                                                                     |
| **模板** | [`templates/architecture.md`](templates/architecture.md)                                                          |
| **要求** | ① 架构概览（分层架构图）<br>② 技术选型及理由<br>③ 系统交互拓扑（Mermaid）<br>④ 关键设计决策（决策点 + 理由说明） |

### 阶段三：详细落地 (Implementation Detail)

**目标**：定义具体的数据存储方案与接口契约。

#### Step 3.1 — 数据库详细设计

| 项目     | 内容                                                                                                              |
| -------- | ----------------------------------------------------------------------------------------------------------------- |
| **产出** | `docs/design/database_design.md`                                                                                  |
| **模板** | [`templates/database_design.md`](templates/database_design.md)                                                    |
| **要求** | ① 完整 ER 关系图（Mermaid erDiagram）<br>② 数据字典（字段名/类型/长度/必填/默认值/备注）<br>③ 完整建表 DDL 语句  |

**验收标准**：

- [ ] 每张表包含 `id`, `create_time`, `update_time`, `is_deleted`, `version` 五大基线字段
- [ ] 枚举字段在备注中列出所有值含义
- [ ] 金额字段标注单位

#### Step 3.2 — 接口详细设计

| 项目     | 内容                                                                                                                          |
| -------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **产出** | `docs/design/api_design.md`                                                                                                   |
| **模板** | [`templates/api_design.md`](templates/api_design.md)                                                                          |
| **要求** | ① 接口定义（URL/Method/描述/鉴权）<br>② 请求参数表（字段/位置/类型/必填/说明）<br>③ 响应参数表<br>④ 请求与响应 JSON 示例     |

**验收标准**：
- [ ] 所有接口响应遵循 `code/message/data` 标准包装
- [ ] 每个接口包含至少一个请求和响应示例

---

### 阶段四：闭环一致性 (Consistency Loop)

**目标**：确保所有文档之间逻辑自洽、术语统一、无遗漏。

#### Step 4.1 — 全局一致性自检与修正

| 项目         | 内容                                                                                                                                              |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| **执行逻辑** | 强制调用校验机制，逐项核对 [`references/consistency_checklists.md`](references/consistency_checklists.md) 中的所有 checklist 项。                  |
| **行动**     | 在输出最终文档前，自动修正所有文档中的冲突与遗漏，确保逻辑闭环。                                                                                  |

**执行检查项**：

1. **需求 vs 架构 (Scope Check)**
   - `requirements.md` 中的所有功能模块是否都在 `architecture.md` 中有对应组件承载？
   - 架构设计是否包含满足 NFR 的具体技术方案？

2. **接口设计 vs 规范 (API Check)**
   - URL 是否遵循 `api_rules.md` 中的资源命名规范？
   - 所有接口 Response 是否包裹了 `code/message/data` 标准结构？
   - HTTP Method 使用是否正确？

3. **数据库设计 vs 规范 (DB Check)**
   - 每张表是否包含五大基线字段？
   - 表名/字段名是否符合命名规范？
   - 是否存在无索引的外键查询或冗余的联合索引？

4. **详细设计 vs 总体设计 (Detail Check)**
   - 核心概念术语是否全文统一？
   - API 输入参数是否能在数据库设计中找到对应存储字段？

---

### 阶段五：系统总纲 (System Master Plan)

**目标**：生成一份"渐进式披露"风格的总纲文档，作为整个文档体系的入口。

#### Step 5.1 — 生成系统总纲

| 项目     | 内容                                                                                                 |
| -------- | ---------------------------------------------------------------------------------------------------- |
| **原则** | **渐进式披露 (Progressive Disclosure)** — 先展示高层概貌，再通过索引引导至细节。                      |
| **产出** | `docs/system_overview.md`                                                                            |
| **模板** | [`templates/system_overview.md`](templates/system_overview.md)                                       |

**结构要求**：

| 章节                                      | 内容要求                                                                                       |
| ----------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **A. 系统定位 (Positioning)**             | 一句话定义系统的核心价值与边界。                                                               |
| **B. 核心设计思想 (Core Philosophy)**     | 系统的"灵魂"（如 DDD、存算分离、插件化等），以及模块拆分设计与关系结构。                       |
| **C. 关键点设计概述 (Key Design Highlights)** | 列出 3-5 个技术难点及其对策（如"高并发策略"、"安全性设计"），让读者不看代码也能掌握核心。     |
| **D. 测试规范与策略 (Testing Conventions)**   | 明确单元测试/集成测试规范及 TDD 指引，包含 Mock 策略、组件依赖及命名规范，指导后续 Agent 开发。 |
| **E. 渐进式文档索引 (Document Map)**      | 采用"标题 + 20~60字核心摘要 + 文件引用链接"的格式，对以上所有文档进行索引。                    |
