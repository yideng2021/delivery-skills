# 工程探索设计 · 棕地认知层（CodeGraph + GitNexus）

> 目标：结合 CodeGraph 与 GitNexus 两个代码知识图谱工具，对棕地（既有）项目的业务、架构、代码逻辑做 AI 梳理，
> 为后续 AI 编程（spec_wf 规约流水线）提供准确、高质量的检索与输入支持。
> 本文记录本轮设计的结论与依据。试点仓：`cnooc_hxcb_requirement-manage-service`（Spring Boot Java 招采采购需求管理）。

---

## 一、问题定位

spec_wf 是规约驱动开发流水线（proposal→spec→design→tasks），天然假设"白纸起步"。
棕地项目的真问题是：**写规约前，得先有一份可信、可检索的"现状认知"**。
CodeGraph + GitNexus 正是补这层"现状理解层"——它们是喂给 spec_wf 的输入源，而非平行的另一套东西。

→ 结论：做一个 **spec_wf 的前置"棕地认知层"**，而不是缝合两个工具。

## 二、工具分工

| 维度 | 主力 | 理由 |
|---|---|---|
| 代码逻辑（调用链、影响面、动态分发） | **CodeGraph** | `explore` 端到端连通 flow（含 callback/Kafka/feign 等静态断链），返回 verbatim 源码，token 成本低 |
| 架构（模块边界、分层、聚类） | 两者 | GitNexus 社区检测自动切模块群；CodeGraph 给分层与影响 |
| 业务（业务流程、入口、契约） | **GitNexus** | 社区/Process/契约（shape_check、api_impact），可做跨域一致性 |

一句话：**CodeGraph 管"代码怎么跑 + 准确源码"，GitNexus 管"业务是什么、改了会炸哪"。** 双源**交叉验证**置信度高于单源。

## 三、核心设计

### 3.1 双消费者模型（关键认知）

认知层有两个不同消费者，需两种产物：

| 消费者 | 产物 | 生命周期 | 产出 skill |
|---|---|---|---|
| **带 MCP 的 agent**（spec_wf writers） | **薄基线**（地图 + 索引 + 活查询出口） | 活查询为主、少固化 | `brownfield-onboarding` |
| **人**（学习 / 交接 / 介绍） | **厚导览**（自洽完整、可通读） | 快照、可重生成 | `architecture-guide` |

二者不冲突：人不会边读边查 MCP，故厚导览正是其所需。

### 3.2 薄基线 + 活查询模型（经批判性审查定型）

- **永久层（薄·人审）**：只放 MCP 给不出的——业务域划分、主线叙事、人审背书 + 回链锚点。
- **活查询层（即时·不固化·源码可信）**：明细每次从 CodeGraph verbatim 源码现取现解读，用完即弃。

> 之所以"不固化 AI 叙事"：固化会把潜在幻觉冻成永久文件，随 commit 腐烂——
> **"错误基线比无基线更危险"**。不固化叙事、只固化人审过的薄骨架，错误就没有沉淀载体。

### 3.3 核心纪律（三条铁律）

1. **机器给"大/全"，AI 判"重要/可读"**：社区按规模排序会被伪域带偏（`Impl(2930)` 是最大伪域），须按业务语义二次过滤。
2. **双源交叉验证**：业务域 = GitNexus 社区 ∩ 包结构；单源不下结论。
3. **断言必回链**：任何结论带 `文件:行号`，可一键核对源码。

## 四、两个 skill

### 4.1 brownfield-onboarding（薄 / 快 / agent）

- **前置完整性闸门（强制首步）**：`codegraph_status` + `list_repos` 三态裁决
  `PASS` / `CONFIRM`（降级，暂停人工确认）/ `BLOCK`（缺失，停止转人工）——**绝不在残缺索引上继续**。
- **6 步流程重建法**：选入口（Controller 写操作）→ `codegraph_explore` 抓链 → 识别驱动机制 → 抽状态机 → AI 即时翻译 → 回链校验。
- **产物**（目标仓 `.brownfield/`）：`00-overview` / `01-architecture`（业务域地图·人审）/ `02-business-flows`（流程索引·人审）/ `05-retrieval-guide`。
- **接 spec_wf**（仿 RBK，零命令耦合）：proposal §0 既有资产盘点、design §5 ADR 复用三问、tasks 影响拆分——writer 在这些节点主动查询本层（已在两个 writer 的 reference 加"棕地基线优先/否则回退 grep"指针）。

### 4.2 architecture-guide（厚 / 慢 / 人）

- **消费** 01 地图，对核心域逐个深描业务主线，产人读完整导览。
- **三条设计**：
  - **分离不污染**：不改薄 01；深描置 `.brownfield/architecture/`。
  - **分级深描**：核心主线域 → 各成 `domain-*.md`（深）；支撑/横切域 → 进 `00-index.md`（浅）。
  - **快照可重生**：回链 + pin commit + 头部声明 + 按域 `detect_changes` **只重跑受影响域**。
- **复用不重造**：复用 onboarding 的前置闸门与 6 步法，本 skill 只做"域级编排 + 拼装叙事"。
- **额外收益**：厚导览可充当 spec_wf design-writer 的 `architecture_refs.path` 来源。

### 4.3 上下游关系

```
brownfield-onboarding（薄/agent）──01 地图──▶ architecture-guide（厚/人）
        共享：前置闸门 · 6 步法 · 回链纪律
下游可选：厚导览 → design-writer 的 architecture_refs
```

## 五、试点验证（cnooc 仓）

- 双索引就绪：CodeGraph 38558 节点 / 76901 边 / 1638 文件；GitNexus 922 社区 / 300 process。前置闸门 `CONFIRM`（embeddings=0，无语义检索，人工确认后继续）。
- **B2 架构提炼**：922 社区 ∩ 20 真实包 → 收敛 **10 顶层业务域**；剔除 `Impl(2930)` 等伪域。分层 Controller(110)/Service(454)/Dao(153)/Entity+Domain/System，基础设施 MySQL·Redis·Kafka·Nacos·OpenFeign·Sentinel。
- **B3 业务流验证**：以"需求提报"跑通——机器标签 `ProcessAfter → Get` 不可用，AI 重建出"双状态机 + 8 审批回调 + SAP/Kafka/仓储跨域触点 + 测试缺口"。证明"入口从 Controller 出发、详情活查询、不用机器 process 标签"的方法成立。
