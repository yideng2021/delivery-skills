## 任务目标
通过代码工程检索、API请求录制数据和前后端代码联合分析，精准还原业务操作流程及其接口依赖关系，输出：
1. **业务流程图**（含主流程、分支节点、条件判断）
2. **各流程节点对应的 API 接口清单**（结构化文本）

---

## 输入材料（分析依据）

请基于以下材料进行综合分析：
- 录制的业务操作脚本（含界面 URL 访问序列）
- 抓取的 API 请求记录（接口路径、请求时序、请求参数概要）
- 工程代码（前端页面组件 + 后端接口实现）

---

## 分析执行步骤

### Step 1：工程检索定位
- 参考 `  @/01-设计过程/04-工程检索/01-search-best-practices.md   ` 中的检索经验
- 根据录制脚本中的**界面 URL** 定位对应前端页面组件/路由
- 根据 **API 请求路径** 定位对应后端 Controller / Service 代码块
- 快速建立"界面 → 前端组件 → API 调用 → 后端处理"的完整映射链路

### Step 2：业务节点识别
- 从操作脚本和 URL 访问序列中，识别关键业务节点（如：列表页加载、详情页打开、表单校验、数据提交等）
- 识别节点间的**顺序关系**与**条件分支**（如：校验通过 → 执行提交；校验失败 → 提示并阻断）
- 识别**并行调用**场景（如：详情页同时调用列表接口 + 基本信息接口 + 字典接口）

### Step 3：节点与 API 接口映射
- 为每个业务节点归集其触发的全部 API 接口
- 标注接口的调用时机（页面初始化 / 用户操作触发 / 条件满足后触发）
- 标注接口间的依赖关系（如：接口 B 依赖接口 A 的返回结果）

### Step 4：结果输出
经过上述分析后，输出最终结果（格式要求见下方）

---

## 输出格式要求

### 一、业务流程总体描述
用 3~5 句话简洁描述该业务操作的整体流程逻辑和目的。

### 二、业务流程图
使用 Mermaid flowchart 语法输出流程图，要求：
- 标注所有主流程节点
- 标注条件分支（如：校验通过/失败）
- 标注并行调用节点
- 节点命名使用业务语义（非技术术语）

```mermaid
flowchart TD
  ...
```

### 三、流程节点 API 接口清单

按节点逐一列出，格式如下：

#### 节点 N：{节点业务名称}
- **触发时机**：{页面初始化 / 用户点击XX按钮 / 条件满足后自动触发}
- **调用接口**：

| 序号 | 接口路径 | 请求方式 | 用途说明     | 调用条件/依赖       |
| ---- | -------- | -------- | ------------ | ------------------- |
| 1    | /api/xxx | POST     | 校验业务规则 | 用户点击提交时触发  |
| 2    | /api/yyy | POST     | 保存工单数据 | 接口1校验通过后触发 |

---

## 约束说明

- **准确性优先**：所有节点和接口必须有代码或录制数据作为依据，禁止推测补全
- **业务语义优先**：节点命名和描述使用业务术语，不暴露不必要的技术实现细节
- **完整性**：不遗漏任何在录制过程中被触发的 API 接口（包括字典、权限、配置类辅助接口）
- **简洁清晰**：最终输出结构层次分明，可直接用于技术文档或需求评审
