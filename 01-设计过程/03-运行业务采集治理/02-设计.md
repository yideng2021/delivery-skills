## 新 Skill 设计思路：`spec-from-flow`（基于运行时流程地图生成生产级业务规约）

### 定位与职责边界

- **上游输入**：`runtime-flow-mapper-skill` 产出的 `flow-map.md`（已有三证对齐、节点清单、状态流转等）
- **下游对标**：`spec-writer-skill` 的 `specs.md` 标准（L0-L4 分层规约），与`spec-writer-skill`没有直接关系，只是参考他的specs输出标准
- **核心差异**：flow-map 是"运行时证据快照"（what happened），spec 是"业务规约"（what SHOULD happen）。本 skill 负责**从证据推导规约**

### 关键设计思路

** 代码深探补全（核心增值）**

- flow-map 只做"落锚"（定位到哪个 handler/service），本 skill 需深入代码逻辑：
  - 读 Service 层实现 → 提取业务校验规则（INV-x）
  - 读状态机/枚举 → 补全完整状态流转（非单样本推断）
  - 读前端表单校验 → 提取前端约束与字段必填规则
  - 读 DTO/VO → 提取业务实体属性完整清单
- 工具：codegraph_explore + codegraph_callees 沿调用链深入

**3. 分层映射规则**

| flow-map 板块     | → spec 层级     | 转换逻辑                                  |
| :---------------- | :-------------- | :---------------------------------------- |
| 流程简介          | L0 业务上下文   | 扩展为业务目标+关键场景+禁区              |
| 节点清单          | L1 用户故事     | 每个核心节点→1-N条 US                     |
| 三证对齐+代码深探 | L2 实体+INV     | 从 handler→service→DAO 提取实体属性与规则 |
| 状态流转+代码验证 | L2 业务流转     | 从枚举/常量补全完整状态机                 |
| 节点交互逻辑      | L3 EARS+Gherkin | 从代码校验逻辑推导正常/异常场景           |
| 全部 AC/INV       | L4 DoD          | 逐条闭环                                  |

**4. 工序设计（草案）**

```javascript
⓪ 输入验证  → ① 业务上下文提炼(L0) →� 代码深探(补全flow-map盲区)
→ ③ 实体与规则提取(L2) → ④ 行为规约撰写(L3) → ⑤ DoD闭环(L4) → ⑥ 自校验
```



**5. 与 spec-writer 的关系**

- 本 skill 不替代 spec-writer，而是**为其提供"从运行时逆向工程"的输入路径**
- spec-writer 从 proposal（正向设计意图）→ spec
- 本 skill 从 flow-map（逆向运行时证据）→ spec
- 产物格式完全对齐 spec-writer 的 template，保证下游（design-writer/task-decomposer）无缝消费

**6. 关键挑战与约束**

- 单样本局限：运行时只录到一条路径，需代码探索补全分支（标 `~inferred`）
- 规则完整性：flow-map 只有接口级信息，INV 需要深入 Service 层提取
- 业务语言纯净：spec 禁用技术术语，需从代码实现"翻译"为业务语言
- 增量标注：对已有系统，所有项应标注为 `[已有·仅引用]` 或 `[已有·扩展]`
