# 现状业务规约产物规格（`{name}_spec.md`）

> 主产物结构（借 spec specs.md 的 L0–L2）+ 业务语言纯净规约 + 校验清单。骨架见 [`../templates/runtime-spec.md`](../templates/runtime-spec.md)。
> **只取 L0–L2**：不写 L3 EARS+Gherkin / L4 DoD / DMN / change_mode / AUTH-id / 增量标注——那些是变更驱动的正向设计机制，与现状逆向无关。

## 顶部强制声明

文件首行（标题前）必须有：

```
> 现状(as-is)业务规约 · 非规范性 · 仅供参考
> 来源：docs/runtime-flow/{name}/{name}_flow-map.md + 前后端代码深探 ｜ 生成时间：{ISO}
```

## 三个板块（L0 / L1 / L2）

### L0 业务上下文

- **业务目标**：1 句业务价值（从 flow-map 简介提炼）。
- **关键场景**：1–3 条典型用户场景（从节点序列归纳）。
- **依赖业务能力（上下文依赖）**：本流程依赖的其他能力/系统，每条带来源标。外部系统（flow-map 外部系统板块）归此，注「外部·本地无源码」。
- **关键术语**：业务概念 = 业务定义（用业务语言，不抄类名）。

### L1 用户故事

每个核心业务节点 → 1–N 条 US：

```
- US-1 `[码][runtime]`：作为 {角色}，我希望 {目标}，以便 {价值}。
  - 溯源：flow-map 节点N / `XxxController.method()`
```

> 纯外部"系统准入"类节点不展开 US，一句概述即可。

### L2 业务实体与规则

**业务实体**（来自 DTO/VO/Entity）：
```
- 实体「{业务名}」`[码]`：1 句业务含义
  - 关键业务属性：{属性1}、{属性2}（业务语义）
  - 溯源：`XxxDTO` / `XxxVO`（字段名进此注，不进上面正文）
```

**业务规则 INV**（来自 Service 校验/分支）：
```
- INV-1 `[码]`：{一条现状业务约束，陈述为"系统当前强制…"}
  - 溯源：`XxxService.validate()` 文件:行
- INV-2 `~inferred: 疑似技术兜底`：{…}
```

**业务流转**（来自枚举 + flow-map 状态流转，按需）：
```
- 状态全集（枚举 `XxxStatusEnum`）：草稿 / 已提交 / 通过 / 驳回
- 迁移：
  - 草稿 → 已提交 `[码][runtime]`：触发 {业务事件}
  - 通过 → 驳回 `[码]`：代码可见、本次运行未覆盖
```
> 仅当实体存在 ≥ 2 业务阶段时启用；mermaid `stateDiagram-v2` 可选，**禁 classDef/style**。

## 业务语言纯净（硬约束，呼应不变量 4）

正文（实体名/属性/规则/术语）必须是**业务语言**；技术细节只许进「溯源」注。

| 禁入正文（技术术语） | 改写为（业务语言） |
|----------------------|---------------------|
| `flowStatus`, `status=2` | "流程状态：已提交" |
| `SupplierDTO.riskLevel` | 实体「供应商」的"风险等级"属性 |
| `checkHighSupplier()` | "高风险供应商校验" |
| HTTP 400 / 异常类名 | "校验不通过被拒绝" |

> 翻译不确定时标 `~inferred`，不硬译。
