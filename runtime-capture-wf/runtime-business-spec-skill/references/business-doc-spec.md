# 业务流程说明 产物规格

> 单份端到端文档 `{name}_业务流程说明.md`，落于 `docs/business-flow/{name}/`。读者是**业务人员**。
> 借 L0-L2 内核（INV/实体/状态机/现状≠规范/语言纯净/术语表），**弃其外形**（用户故事轴/技术编号/溯源锚/层级标签）。

## 一、frontmatter（轻量）

```yaml
---
name: {name}
source_inputs: [docs/prd/{name}/, docs/runtime-flow/{name}/{name}_flow-map.md]   # 溯源回查入口
generated: {ISO}
---
> 现状业务流程说明 · 描述系统当前实际运转 · 非规范
```

> 不内置 sidecar/溯源锚；需深溯源时回 `source_inputs`（ui-collab 文档 + flow-map）查。

## 二、正文结构（五段）

```
# {业务流程名} — 业务流程说明

## 一、业务概述
{谁在什么场景下做什么、达成什么业务目标}
**业务术语**：{对齐黑话，3-8 条}
**依赖的外部业务能力**：{外部系统翻成业务能力，如"风控评估""附件存储"}

## 二、业务流程（端到端）
按「角色 + 环节」时序描述；配 Mermaid 流程图。

## 三、各环节业务规则
### 环节N：{业务动作名}（角色：{谁}）
- 前置条件：…
- 业务规则：
  · {精确规则一句}〔已核实〕
  · {精确规则一句}〔待核实〕
- 产出数据：{实体/状态变化}
- 异常处理：…

## 四、业务数据流转
核心业务实体在各环节的状态变化（状态机，业务语言）。

## 五、角色与权限
谁能做什么（权限多为〔待核实〕，继承 ui-collab）。
```

## 三、组织轴红线（防退化为规约）

守三道闸，否则会变成"换皮 spec-mapper"：

1. **组织轴 = 流程环节**（不是用户故事、不是实体清单）。
2. **读者 = 业务**（按业务人员能读来写）。
3. **零术语 + 确信度词**（每条事实/规则可信度业务化）。

## 四、Mermaid 约定

- 节点 = **业务动作**（"提交申请""审批""生成订单"），**禁**接口/路径/技术事件。
- 只保留结构，禁 `classDef`/`style`/`class`。
- 角色可用 `subgraph` 泳道表达（可选）。

## 五、INV 纪律（规则怎么写）

- 规则写成**精确、可判定**的一句，挂在业务实体上（详见 [`code-deepdive.md`](code-deepdive.md) §3）。
- **不带技术编号**（无 `INV-1`）、**不带溯源锚**（无 `类.方法`）。
- 每条规则**必带确信度词**。

## 六、确信度词

| 词 | 含义 |
|----|------|
| 〔已核实〕 | 代码确证 / 人工已确认，可信 |
| 〔待核实〕 | 单样本 / 推断 / 未覆盖分支 / 未梳理界面，待人核 |

> 两档。表达不确定性用业务语言（如"基于当前观察，可能还有其他情况"），不用技术标注。

## 七、自校验（产出前必过）

- [ ] **零术语扫描**：正文无 `/api/`、`http`、`.java`/`.vue`/`.js`、类名（`XxxController`/`XxxService`）、字段名（驼峰技术名）、`[码]`/`[runtime]`/`~inferred` 等标注黑话。
- [ ] 每个环节有：前置 / 业务规则（≥1，带确信度词）/ 产出数据。
- [ ] 流程闭环且按 `nav_context` 串；未梳理界面已标〔待核实·未梳理〕。
- [ ] 数据流转覆盖核心实体状态变化；角色与权限齐。
- [ ] 顶部含「现状·非规范」声明。

## 八、目录与命名

- 目录：`docs/business-flow/{name}/`（不存在则建）。
- 命名：`{name}_业务流程说明.md`，单份。
- 写入：`write_file` 直写工作区内目录。
