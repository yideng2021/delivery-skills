# 代码深探法（核心增值）+ INV 纪律

> 沿 flow-map 代码锚深入 Service 层抽业务规则。**到 Service + 校验 + 状态机即止，不下 DAO/SQL。**
> 这是 ui-collab 故意没做的"下 Service 抽规则"，由本 skill 一次性补上（唯一规则源）。

## 一、深探路径（每环节走一遍）

flow-map 三证表已给每个界面操作的后端 Handler 锚点。从锚点出发：

```
Handler（入口，flow-map 已锚）
  → callees / 调用链：Service 主逻辑     ← 抽：校验、计算、状态转换
  → DTO / VO                            ← 抽：业务实体 + 业务属性（用业务名）
  → 枚举 / 常量                          ← 抽：完整状态机（取全部状态值）
  → 前端校验（flow-map 前端锚）           ← 抽：字段级约束（必填/范围/格式）
  ── 到此为止，不进 DAO / Mapper / SQL ──
```

- **检索遵规范**：`codegraph_explore`/`callees` 优先，多仓带 `projectPath`/`repo`；落空再 grep。
- **状态机取全**：枚举里所有状态值都取，不止运行时见过的——这是代码相对 runtime 的增值（〔已核实〕）。
- **运行时未覆盖的分支**：代码可见但本次没跑到 → 规则成立但标〔待核实〕（单样本/未覆盖）。

## 二、抽什么（三类业务规则）

| 类型 | 从哪来 | 示例（业务语言） |
|------|--------|------------------|
| **校验规则** | Service 入口校验 + 前端校验 | "金额必须大于 0""标题不可为空" |
| **计算规则** | Service 计算逻辑 | "总价 = 单价 × 数量 − 优惠" |
| **状态转换** | 枚举状态机 + 流转条件 | "审批通过后状态变为『已生效』，且不可再编辑" |

## 三、INV 纪律（怎么把规则写好）

借 spec L0-L2 的 **INV（业务规则=一条永真约束）**，但**弃其外形**：

- **借**：规则写成**精确、可判定**的一句，挂在**业务实体**上。
- **弃**：不带技术编号（不写 `INV-1`）、不带溯源锚（不写 `类.方法`/`文件:行`）、不套用户故事结构。

写法对照：

| ✅ INV 纪律（精确） | ❌ 含糊散文 |
|------|------|
| "订单金额必须大于 0，否则不可提交" | "金额要合法" |
| "审批通过后订单不可再修改" | "审批之后一般就不能改了" |
| "退款金额不得超过原支付金额" | "退款有上限" |

## 四、确信度归档

| 词 | 归入条件 |
|----|----------|
| 〔已核实〕 | 代码校验/枚举直接确证；或 ui-collab `[人工]`/`[runtime+码]` 已确认 |
| 〔待核实〕 | 单样本观察、运行时未覆盖分支、疑似防御兜底/遗留逻辑、ui-collab `~待确认`、未梳理界面 |

## 五、冲突优先级

- 界面/操作事实（谁触发、做什么）：**ui-collab 人工确认最高**。
- 规则细节（校验阈值、状态条件）：**代码深探为准**。
- 二者冲突 → 人工确认优先，并把代码侧差异标〔待核实〕提示人核。

## 红线

- **不下 DAO/SQL**：深探到 Service+校验+状态机即止。
- **不造规则**：代码里没有的约束不写；推测的标〔待核实〕，不当确证。
- **现状≠规范**：只述"当前强制 X"，不写"应当 X"。
- **技术证据不入正文**：类名/字段名/路径只用于内部定位，正文只出业务语言。
