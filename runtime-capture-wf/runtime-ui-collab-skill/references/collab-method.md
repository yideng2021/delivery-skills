# 逐界面人机协同法（核心能力）

> 每个界面走一遍 S1→S2→S3 循环，**确认后即时落盘**。本文件给协同三步的操作口径与红线。
> 标注词表与升档自检见 [`ui-doc-spec.md`](ui-doc-spec.md)。

---

## ① 界面范围确认（进入循环前一次）

1. 读 flow-map 节点清单 + 三证表，**以 session_log 的 route/URL 为主键**推断「节点 → 界面」映射。
   - 节点↔界面是 **N:M**：一节点可能是子步骤、多节点可能是同界面不同状态。
2. 展示界面清单表格，用户确认/修正：

| 界面ID | 界面名称 | route | 类型 | 对应 flow-map 节点 | 触发入口 |
|--------|---------|-------|------|-------------------|---------|
| UI-01 | … | /xxx/list | 独立/合并/子界面/多视角 | 节点N | … |

3. 确认后逐界面进入 S1。

---

## S1 — 草稿生成

**按界面切片读取**：flow-map 为索引定位本界面对应节点/route，再据此**只读** records.ts / session_log / api_details 中
本界面相关片段（api_details 按需），不整批灌入，契合 capture 两段式加载。

草稿四块内容：
- **业务介绍**：是什么 / 谁用 / 能做什么 / 关键约束
- **上下文说明**（五维）：前置 / 后置 / 数据 / 权限 / 状态（前置/后置取自 session_log 导航序列）
- **功能地图**：骨架树形 + 每功能点描述，**按界面操作/功能点组织**（触发方式/操作序列/字段取自 records.ts）——区别于 spec 的角色故事切法
- **界面流程图**（Mermaid）：结构取自界面内操作时序

> 草稿中所有未确证内容先标 `~inferred` / `~unresolved`，进入 S2 收敛。

---

## S2 — 推断收敛（三级，不可越级）

逐一处理草稿中的 `~inferred` / `~unresolved`：

**1. runtime 原始证据自证**（最优先）
查 records.ts / session_log / api_details——多数"触发方式 / 操作序列 / 字段"在此直接命中，标 `[runtime+码]` / `[runtime]`。
- **跨服务 0 命中守卫**：代码回查 0 命中**不等于无此逻辑**。调用已被 session_log/api_requests 证实 → 标 `[runtime]` + 注"代码不在本工程（疑外部/微前端/共享组件）"，**禁止下'无此逻辑'结论**；与 flow-map 已有 `[外部:xxx]`/`~unresolved` 裁决不冲突。
- **时序≠因果守卫**：相邻接口不等于因果。要断言"因果/存在中间步骤"时，先 diff 同名接口多次请求体的字段变化佐证；佐证不足 → **只记时序、不标因果**。非断言因果时不强制逐条 diff。

**2. 代码回查补全**
runtime 未覆盖的，用 flow-map 锚点浅回查代码自证（到锚点验证为止），升格标注。
- **缺证不断言**：回查无源码佐证的行为/规则，标 `~待确认`，**不直接写为定论**。

**3. 问人兜底**
仅**业务判断类**（给谁用 / 业务意图 / 业务边界）回查无法定论的，才向用户追问：
- 问人**已定** → 升 `[人工]`，原 `~inferred` 移除
- 问人**未定**（用户选"保留标注"）→ 转 `~待确认` 显式残留

### 追问呈现规则

- **复杂推断**（业务意图、边界规则、runtime+代码均未定论）→ 卡片式：

```
┌─ ❓ 待确认 ──────────────────────────────────────────┐
│ 问题：[功能X] 面向哪类角色 / 业务边界                │
│ 我的推断：仅审批人可见此操作                         │
│ 推断依据：records.ts 单样本 + 代码无显式权限校验     │
│ 推荐答案：是                                         │
│ 可选：[是] [否，实际是______] [不确定，保留标注]     │
└──────────────────────────────────────────────────────┘
```

- **简单确认**（字段名、操作文案）→ 对话式直接问。

> 红线：**只问 runtime 与代码都答不了的业务判断**。「触发方式=点编辑按钮」这类 records.ts 已录到的，直接读出标 `[runtime+码]`，不许当推断去问。

---

## S3 — 用户答 → 升格标注 → 即时写盘 → 确认菜单

1. 据回答更新草稿与标注（`~inferred` → `[人工]`），**立即写入该界面 md（`status: draft`）**。
2. 展示确认菜单：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  UI-01 · [界面名称] — 草稿已生成 / 已落盘
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  [1] 全部确认，进入下一界面
  [2] 修正业务介绍 / 上下文
  [3] 修正功能地图
  [4] 补充遗漏功能
  [5] 跳过此界面
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

3. 菜单分流：
   - `[1]` → 过[升档自检](ui-doc-spec.md#升档自检)，通过则置 `status: confirmed` + `confirmed_at`，进入下一界面
   - `[2]/[3]/[4]` → 回 S2 局部修订后重展菜单
   - `[5]` → 置 `status: skipped`，进入下一界面

> **即时落盘**替代收尾统一输出：每界面即写盘，收尾只汇总。
> **断点续做**：重启扫 `docs/prd/{name}/` 各文件 `status`，`confirmed`/`skipped` 跳过、`draft` 续做。

---

## 收尾汇总

所有界面处理完毕，扫 `status` 汇总：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  所有界面协同完成（均已落盘，按 status 统计）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  confirmed：UI-01 / UI-02
  skipped  ：UI-03
  draft 残留：无
  产物目录：docs/prd/{name}/
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> `~待确认` 不设硬上限——诚实记录未决即可，各界面 `confidence.待确认` 数在收尾透明展示给人。
