## 一、定位：棕地领域的 critic

一句话：**`brownfield-impact-analyzer` 之于「改既有代码」，如同 `spec-critic` 之于「审规约」**——一个按需调用、产咨询件、可独立可嵌入的专项分析器。

| 维度 | 内容 |
|------|------|
| **它是什么** | 给定「改动意图 + 代码库」，产出**冲突点 + 影响面 + 侵入/接缝建议 + 回滚设计 + 特征测试登记**的分析器 |
| **它产出什么** | `impact.md`——**咨询性参考件**，不进主 schema 校验（先例：`critic.md`） |
| **它不是什么** | 不是流水线阶段、不是新文档治理类型、不替代 proposal/spec/design；不抢编排权 |
| **谁调用它** | proposal-writer（§0 发现业务冲突/重构时）、spec-writer（`impacted_modules` 含既有重构/关键接口调整时）、用户（独立调用） |

## 二、与 v1 / v2 的关系（为什么 v3 更克制）

| 维度 | v1 | v2 | **v3（本稿）** |
|------|----|----|--------------|
| 形态 | 新阶段 + 新文档 cip.md | proposal 的模式增强 | **独立分析器 skill（critic 范式）** |
| 对现有 skill 改动 | 大（新阶段接入） | 大（6 字段 + I-G + 模板 §0.4） | **极小（各加一行触发提示）** |
| frontmatter 新增必填字段 | 10 | 6 | **0**（impact.md 用自有局部 frontmatter，不进主 schema） |
| validate.mjs 改动 | 改 4 处 + 新 schema | 扩 1 个 $defs + I-G1~6 | **0** |
| CG | 双 CG | 单 CG + 棕地题库 | **复用现有 CG，不新增题库强制** |

> v3 的取舍逻辑：棕地分析的真正价值是「**早一步看清会撞到哪、怎么安全退**」，这是**分析能力**，不是**治理流程**。用最轻的形态（咨询件）交付能力，把强制收敛到唯一必要的一处（改既有方法时的安全门）。

## 三、skill 形态（对齐 spec-critic）

### 3.1 触发（三入口，与 critic 同构）

1. **proposal-writer 按需调用**：§0 既有资产盘点中发现「与既有功能的业务冲突」或「需重构/替换既有能力」时。
2. **spec-writer 按需调用**：`impacted_modules` 中存在「既有重构、关键接口调整等影响现有工程」的情况时。
3. **用户独立调用**：「分析一下这个改动的冲突和影响」/「这块 bug 修复会撞到哪」。

### 3.2 角色边界（抄 critic 的克制）

- **不改任何 writer 正文**：只产 `impact.md`。
- **不抢编排权**：被调用、给结论，是否采纳由调用方决定。
- **零命令名硬耦合**：proposal/spec-writer 通过「读 impact.md 文件」消费结论，不调用本 skill 的内部能力。

### 3.3 非目标（划清边界，防膨胀）

- 不生成规约/设计/任务文档；
- 不维护需求账本；
- 不做组织级治理（不替代 CODEOWNERS / PR review policy）。

## 四、产物 `impact.md` 结构（≤2 页，专业简洁）

```yaml
---
# 局部 frontmatter，仅本 skill 读写，不进主 schema 校验（同 critic.md）
change_ref: {change_name 或 一句话改动标识}
analyzed_at: <iso8601>
route: direct | spec_chain          # 路由建议：轻量直交 / 走规约链
invasion_tier: new_file | composition | inheritance | decorator | modify_existing
modifies_existing: true | false     # 是否触碰既有方法（决定安全门是否生效）
has_conflict: true | false
safety:                             # 仅 modifies_existing == true 时必须填实
  characterization_tests: filled | n/a
  rollback: filled | n/a
  confirmed: done | pending
---

# 1. 改动意图（≤2 句）

# 2. 冲突点
| 冲突对象 | 类型 | 与既有的关系 | 处理建议 |
|---------|------|-------------|---------|
| `{既有功能/接口}` | 语义冲突 / 接口变更 / 在途PR | 替换 / 共存 / 废弃 | … |

# 3. 影响面
| 维度 | 现状 | 变更后 | 影响范围（调用方/数据/接口/配置） | 风险 |
|------|------|--------|----------------------------------|------|

# 4. 侵入与接缝建议
- 推荐侵入阶梯：`{invasion_tier}`（理由：…）
- 接缝点：`{构造器注入 / 工厂替换 / 接口扩展 / 子类覆写}`（位置：…）

# 5. 回滚设计（modifies_existing 时必填）
| 开关 key | 默认值 | owner | expire_after |   ← expire_after 不得为空

# 6. 特征测试登记（modifies_existing 时必填）
| 受改方法 | 路径 | 需 pin 的原有行为 |   ← 仅登记需求，测试代码由下游写

# 7. 路由建议
- [ ] route: direct —— 小改动、安全三件套齐 → 直接交付代码 skill
- [ ] route: spec_chain —— 跨团队/替换废弃/破坏性 → 走 proposal→spec→design→tasks
```

## 五、三触发点的轻量嵌入契约（核心要求：尽量少改）

**原则：proposal-writer / spec-writer 各只加一行「按需触发提示」+ 一处「被动引用」，不新增必填字段、不改不变量主轴、不强制调用。**

### 5.1 proposal-writer（§0 发现业务冲突/重构时）

在 [`proposal-writer-skill/SKILL.md`](../../spec_wf/proposal-writer-skill/SKILL.md) 的「文件导航」或 §0 写法处追加一行**提示性**说明（非强制不变量）：

> 若 §0 盘点发现与既有功能的**业务冲突**或需**重构/替换既有能力**，建议先调用 `brownfield-impact-analyzer` 产出 `impact.md`；其 §2 冲突点 / §4 侵入建议可作为 §0.3「既有约束与历史决策」与 §2「关联既有资产」的**素材引用**（被动引用，不强耦合）。

落点复用既有结构：
- `impact.md §2 冲突点` → proposal §0.3 / §2 的引证素材；
- `impact.md §3 影响面` → proposal §2「Blast Radius」列的填充依据（**该列 proposal 模板已有**，无需新增）。

### 5.2 spec-writer（impacted_modules 含既有重构/关键接口调整时）

在 [`spec-writer-skill/SKILL.md`](../../spec_wf/spec-writer-skill/SKILL.md) 的「输入」表或「文件导航」处追加一行**提示性**说明：

> 若 `impacted_modules` 中存在**既有重构、关键接口调整等影响现有工程**的情况，建议调用 `brownfield-impact-analyzer`；其 §3 影响面 / §6 特征测试登记可作为本 spec **增量标注（`[已有·修改]`/`[已有·废弃]`）与 Diff 表 / 迁移路径**的输入素材（被动引用）。

落点复用既有机制:
- `impact.md §6 特征测试登记` → spec 不变量 3 的 `[已有·修改]` 配套 Diff 的输入;
- `impact.md §2 冲突点` → spec `[已有·废弃]` 的迁移路径+兼容期窗口依据。

> 注:proposal/spec-writer 的 CG、增量标注、Diff 机制**全部沿用既有**,impact.md 只是给它们"喂料"。**不新增 CG 题库强制、不新增 frontmatter 必填字段。**

### 5.3 用户独立调用

无需任何 change 上下文即可运行;`impact.md` 可落在 `docs/spec/{change_name}/`(若已有 change)或用户指定路径(独立场景)。

## 六、轻量 vs 规约链：仪式与安全解耦（只分两路，不造多档）

放弃 v2 的三档 `br_tier` 字段(过度设计)。分析器只在 §7 给**两个路由建议**,由调用方/用户裁决:

| route | 适用 | 仪式 | 安全(强制) |
|-------|------|------|-----------|
| **`direct`** | 小改动;无跨团队;无替换/废弃/破坏性 | 跳过 spec/design/tasks,`impact.md` 直接交付代码 skill | **若 `modifies_existing`:特征测试 + 回滚 + 确认 三件套必须齐**(见 §七) |
| **`spec_chain`** | 跨团队 ∨ 替换/废弃 ∨ `[BREAKING]` | 走 proposal→spec→design→tasks 既有全链 | 由既有链的 CG/critic/validator 兜底 |

> **关键准则:轻量化只能减"仪式",不能减"安全"。** 一个 3 行的既有方法内修复可以走 `direct`(不写四份文档),但只要它 `modifies_existing`,特征测试/回滚/确认一个都不能少。这条线把 Feathers 的「改既有代码必须有测试保护」守住。

## 七、唯一硬约束:`direct` 路径的 pre-flight 安全门

这是 v3 **唯一**新增的强制点(替代 v2 的 6 条 I-G 规则),且不落在 validate.mjs、不依赖 change 目录锚点:

> **代码 skill 在动手前必须读 `impact.md`。若 `route == direct` 且 `modifies_existing == true`,则 `safety.characterization_tests`、`safety.rollback`、`safety.confirmed` 三者必须分别为 `filled / filled / done`;任一缺失 → 拒绝执行,回退要求补齐或升 `spec_chain`。**

为什么只留这一道:
- `spec_chain` 路径的安全由既有 CG/critic/validator 已经兜底,无需重复;
- `direct` 路径绕过了全部既有硬门,**这是唯一会"无保护改既有代码"的口子**,必须且只需在此设一道闸;
- 不触碰既有方法(`new_file`/纯新增)时无安全门,零摩擦。

## 八、刻意不做的设计（避免过度复杂化的显式声明）

为兑现「克制」准则,以下 v1/v2 出现过的机制**在 v3 中明确删除**:

| 删除项 | 理由 |
|--------|------|
| 新建 cip.md 文档治理类型 | validator 不可见;改为咨询件 impact.md(同 critic.md) |
| pre-proposal 新阶段 | 用「按需调用的分析器」替代,不进流水线 |
| proposal/spec 新增 6~10 个必填 frontmatter 字段 | 改动过大;impact.md 用自有局部 frontmatter 承载 |

| LOC 行数分档 / `br_tier` 字段 | 用 route 两路 + 安全门替代,更简单也更准 |
| 10 个 references 文件 | 合并为 3 个(见 §九) |

## 十二、边界声明

- **不适用**:greenfield、纯新增文件无冲突(分析器可被调用但快速返回「无既有触达」)、一次性脚本。
- **失效场景**:
  - CODEOWNERS / git blame 不可得 → `impact.md §2` 标「治理债务」,route 倾向 `spec_chain`;
  - 既有代码零测试 → §6 特征测试需先覆盖既有行为,route 倾向 `spec_chain` 并标「测试债务」;
  - route 边界模糊 → 取强约束(`spec_chain`)。
- **咨询件的固有弱点**:`impact.md` 同 `critic.md`,本身不强制被执行;唯一例外是 `direct + modifies_existing` 时由 §七 pre-flight 兜底。
- **诚实声明**:本 skill 提供「分析与检测」,不替代组织级治理;它把棕地的「冲突点 + 影响说明」以最轻形态固化为可复用的参考件,供 spec-design-workflow 关键环节按需取用。
