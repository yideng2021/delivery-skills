# Diff 表与迁移映射 — `[已有·修改]` / `[已有·废弃]` 的承接规则

> 本文件是 spec 阶段对 `[已有·修改]` / `[已有·废弃]` 标注的**配套表格写法**唯一权威。
> 行级增量标注语义见 [`./increment-annotation.md`](./increment-annotation.md)；本文件回答\"标完之后还要补什么表\"。
>
> 三类表按层归口：L2 实体属性 Diff（§1）、L3 行为 Diff（§2）、L3 AC 迁移映射（§3）、L0/L2/L3 废弃迁移（§4）。

---

## §1 L2 实体属性 Diff 表

**触发**：L2 有 `[已有·扩展]` / `[已有·修改]` 标注的业务实体。
**仅 `[已有·扩展]` 时 Diff 表可省略**（新增属性即扩展点本身）；**`[已有·修改]` 强制**。

### §1.1 表格形态

| 实体 | 属性名 | 原定义 | 新定义 | 影响下游 |
|------|--------|--------|--------|---------|
| `{用户}` | `风险等级` | —（不存在） | 枚举：低 / 中 / 高 | 触发 REQ-00x 的分流逻辑 |
| `{订单}` | `金额单位` | 元 | 分 | 既有 INV-x 需同步修订 |

### §1.2 写作铁律

- **业务语言纯净**：列\"原定义 / 新定义\"只写业务概念（枚举值 / 单位 / 计量口径 / 约束语义），**不写**字段类型 / 数据库列类型 / 序列化格式
- **影响下游**列必须给出至少一条具体 INV-x / REQ-xxx / 业务流程编号；填\"无影响\"视为违例（既然语义变化，下游必有人感知）
- 跨实体属性的同源修改用多行表达，**不合并为\"批量修改\"一行**

## §2 L3 行为 Diff 表（针对 `[已有·修改]` REQ）

**触发**：L3 有 `[已有·修改]` 标注的 REQ。
**强制**：必须给出\"既有 REQ 引用\"+ 三维 Diff。

### §2.1 表格形态

| 维度 | 原行为声明 | 新行为声明 |
|------|-----------|-----------|
| 触发条件 | `{原 When 子句}` | `{新 When 子句}` |
| 系统动作 | `{原 Then 子句核心动作}` | `{新 Then 子句核心动作}` |
| 约束 | `{原：时限 / 格式 / 幂等性 / 拒绝路径}` | `{新：时限 / 格式 / 幂等性 / 拒绝路径}` |

### §2.2 写作铁律

- **三维必填**：触发条件 / 系统动作 / 约束 三行必须全部填写；任一行\"无变化\"必须显式写 `（不变）`，**不允许整行省略**
- **既有 REQ 引用**必须含**具名锚点**（如 `specs/auth-base.md#REQ-12`），仅写文件路径视为违例
- 行为 Diff 表与 `[已有·修改]` 的 EARS 句式必须自洽（新 EARS 句即\"新行为声明\"三维的合成）

## §3 L3 AC 迁移映射表（针对 `[已有·修改]` REQ）

**触发**：`[已有·修改]` REQ 下含既有 Scenario 的处置。
**强制**：必须给出 AC 迁移映射，避免既有 AC 静默丢弃。

### §3.1 表格形态

| 既有 AC 编号 | 处理方式 | 新 AC 编号 | 备注 |
|-------------|---------|-----------|------|
| `AC-12-01` | 保留 | `AC-002-01` | 仅编号迁移，断言不变 |
| `AC-12-02` | 修改 | `AC-002-02` | Then 子句增加\"风险评分通过\"前置 |
| `AC-12-03` | 拆分 | `AC-002-03` / `AC-002-04` | 原单场景拆为正常 / 异常两条 |
| `AC-12-04` | 废弃 | — | 原规则被 INV-3 取代，已无对应行为 |

### §3.2 处理方式闭集

| 处置 | 语义 | L4 DoD 增量闭环义务 |
|------|------|--------------------|
| **保留** | 断言不变，仅编号映射 | 必须在\"既有 AC 复测\"中勾选 |
| **修改** | 断言变化 | 既有 AC 不复测，新 AC 进入 L4 功能级 AC |
| **拆分** | 一条 → 多条 | 各拆分项独立进入 L4 功能级 AC |
| **废弃** | 删除断言 | 必须在\"AC 迁移完整性\"中确认代码无残留引用 |

### §3.3 写作铁律

- **零静默丢弃**：表中不允许出现\"既有 AC 不再适用故未列出\"——必须显式列出并标 `废弃`
- **新 AC 编号唯一性**：拆分后的新编号在同一 spec 内不得重复；空集（`废弃`）用 `—`
- **备注必填**：每行至少一句话说明变化点或废弃理由

## §4 `[已有·废弃]` 迁移要素

**触发**：任一 `[已有·废弃]` 标注（L1 US / L2 INV / L3 REQ / L0 合规约束 / L0 关键术语）。
**强制**：两要素 + 一处 L4 闭环。

### §4.1 必填两要素

```markdown
- `[已有·废弃]` **INV-7**：原\"邮箱唯一性\"约束
  - **迁移路径**：由 INV-1（跨租户唯一性）取代；旧用例改写指引见 `docs/migration/inv-7.md`
  - **兼容期窗口**：v2.3 起冻结写入，v2.5 起拒绝，v3.0 删除
```

| 要素 | 内容要求 | 反模式 |
|------|---------|--------|
| **迁移路径** | 至少包含\"由谁取代\"+\"旧用例如何改写或引导到何处\" | ❌ 仅写\"废弃\"不给出后续 |
| **兼容期窗口** | 至少含**起始时机**与**终止时机**（版本号或时间） | ❌ 仅写\"将于未来版本删除\" |

### §4.2 L4 DoD 增量闭环必填

- [ ] `[已有·废弃]` 项：迁移路径已落地（替代项可用 / 改写指引已发布）
- [ ] `[已有·废弃]` 项：兼容期窗口公告已发布（迁移文档 / changelog / 通知）
- [ ] `[已有·废弃]` 项：废弃公告已通知 `impacted_modules` 中列出的下游

## §5 三表的协同关系

```
L2 [已有·修改] 实体  ─触发─▶ §1 实体属性 Diff 表  ─影响下游列─▶ 关联 INV-x / REQ-xxx
L3 [已有·修改] REQ   ─触发─▶ §2 行为 Diff 表 + §3 AC 迁移映射表
L0/L2/L3 [已有·废弃] ─触发─▶ §4 两要素 + L4 DoD 增量闭环段
```

三表之间**互不替代**：实体语义变化必有行为变化 → §1 触发后 §2/§3 一并触发；只填一张表视为违例。

---

## §6 校验规则（供 Stage 4 审计）

- 每条 `[已有·修改]` L2 实体 → §1 表中至少一行（除非紧邻显式写明\"无属性级变化，仅关系语义调整\"）
- 每条 `[已有·修改]` L3 REQ → §2 表 + §3 表**双必填**
- 每条 `[已有·废弃]` → §4 两要素 + L4 DoD 三项闭环
- §3 表中\"处理方式\" ⊆ {`保留`, `修改`, `拆分`, `废弃`} 闭集
- §1 表中\"原定义 / 新定义\"列**不出现** SQL / 字段类型 / 主外键等技术词（业务语言纯净铁律）
