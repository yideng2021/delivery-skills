# DMN 决策模型 — 何时启用 / 如何嵌入 / 如何去重

> 本文件回答 spec L3 行为规约阶段引入 DMN（Decision Model and Notation）决策表的**三件事**：
> 1. **何时启用**（启用判据与不启用判据）
> 2. **如何嵌入**（决策表 / BKM / DRD 三件套的最小写法）
> 3. **如何与其他层去重**（DMN 与 L0 合规 / L2 实体 / L3 EARS+Gherkin / L4 DoD 的红线）
>
> **DMN 规范本体不在本文件展开**（DRD 五类节点、三类箭头、决策表 7 种命中策略、FEEL 语法、BKM 等概念详见外部 DMN 规范文档）；本文件聚焦于\"spec 中怎么用 DMN\"。

---

## §1 何时启用（命中 ≥ 2 条则嵌入）

| 判据 | 含义 |
|------|------|
| ① 多维交叉 | 决策由 **≥ 3 个维度交叉**组合而成（如\"用户等级 × 时段 × 金额\"） |
| ② 业务高频变更 | 规则**变更频率高**，且变更主导方是业务 / 运营 / 产品 |
| ③ 业务可读可改 | 规则需要**业务人员可读可改**（不能埋进代码 `if-else`） |
| ④ 审计追溯 | 规则需要**审计与合规追溯**（金融、保险、医保、风控、动态定价） |
| ⑤ 跨场景复用 | 同一组规则被**多个 REQ / 场景复用** |

**判定逻辑**：命中 **≥ 2 条** 时建议嵌入；命中 ≥ 4 条时**强烈推荐**。

---

## §2 何时**不**启用（强行嵌入即过度设计）

| 反向判据 | 处置 |
|---------|------|
| 单一线性规则、≤ 2 个条件 / ≤ 4 行规则的小决策 | Gherkin 直写，**不嵌入** DMN |
| 规则与流程深度耦合（长事务 / 状态机 / 补偿） | 归 L2 业务流转，**不嵌入** DMN |
| 规则本质是 ML 模型 / 历史时序计算 | 不属 DMN 适用范围，归数据 / 算法 spec |

> **过度嵌入红线**：spec 中每 REQ 都挂决策表 ≠ 体系化；视为模板滥用，audit 拒绝通过。

---

## §3 编号规则

- 决策表统一编号 `DMN-xxx`（`xxx` = spec 文件内单调递增三位数）
- 每行规则编号 `DMN-xxx-Rn`（`n` = 规则号，1 起）
- BKM 编号 `BKM-xxx`
- DMN 编号空间**与 `INV-x` / `AC-xxx-xx` 并行不冲突**，不引入新一级前缀

---

## §4 形态按需取用

DMN 三件套不强制齐全，按需启用：

| 形态 | 启用条件 | 内容 |
|------|---------|------|
| **决策表** | DMN 启用即必有 | 核心载体；每张表**强制声明命中策略**（推荐 `U` 唯一 / `C` 累加；**禁用 `F` 首次匹配**），并**强制补默认兜底行** |
| **BKM**（业务知识模型） | 某计算公式被 **≥ 2 个决策复用** | 抽出为独立 BKM；单次使用应内联到决策表的结论列 |
| **DRD**（决策依赖图） | 存在 **≥ 2 个决策节点 + BKM** 的依赖关系 | 用 Mermaid 画依赖图；单一决策表无需 DRD |

## §5 与其他层的协作与去重（核心）

| 与之关联 | 协作方式 | 反模式（必避） |
|---------|---------|---------------|
| **L0 合规约束** | 决策表\"知识源\"指向 L0 已列出的政策文件，**不重抄政策原文** | 决策表里大段复制法规条文 |
| **L2 实体属性** | 决策表 Input Data 直接引用 L2 业务属性名，**不重新定义** | 决策表里再画一遍实体字段 |
| **L2 业务流转** | 业务阶段跃迁的复杂\"业务守卫条件\"可引用决策表编号（如 `DMN-001`） | 把业务阶段流转塞进决策表 |
| **L3 EARS** | EARS 引出\"调用决策表 DMN-xxx\"，**不重述规则细节** | EARS 把规则用自然语言再讲一遍 |
| **L3 Gherkin** | 只写代表性场景（每命中策略分支 1 条 + 兜底行 1 条 + 整体参数化 1 条） | 为决策表每行规则各写一条 Gherkin |
| **L4 DoD** | 勾选\"决策表 N 行规则均有参数化测试覆盖\" | DoD 不再为每行规则单独写一条 |

> **Gherkin 去重红线（最常违反）**：启用 DMN 时，决策表每行规则即一条规则级 AC；Gherkin **只写代表性场景**（每个命中策略分支 1 条 + 兜底行 1 条 + 决策表整体参数化覆盖 1 条），**不再为每行规则各写一条 Scenario**。

## §6 增量标注规则（`change_mode != greenfield`）

- 每张决策表（如 `DMN-xxx`）开头加增量标注：`[新增]` / `[已有·扩展]` / `[已有·修改]` / `[已有·废弃]`
- **行级标注**：决策表内每行规则若需差异化标注（如本次仅新增 R3、R4），在\"规则号\"列旁加同样的增量标注
- **修改既有决策表**：附「规则迁移表」（既有规则号 / 处理方式 / 新规则号 / 原 → 新差异），与 L3 AC 迁移映射（[`./diff-and-migration.md`](./diff-and-migration.md) §3）同构

## §7 最小化决策表示例（AI 填空参照）

3 维交叉 + 兜底行，体现最小完整形态：

```markdown
#### DMN-001 `[新增]` 订单优先级分流

> **关联 REQ**：REQ-003
> **命中策略**：U（唯一匹配）
> **知识源**：L0 合规约束\"VIP 客户优先级政策\"

| 规则号 | 增量标注 | 输入: 用户等级 | 输入: 时段 | 输入: 金额 | 输出: 优先级 | 业务备注 |
|--------|----------|---------------|-----------|-----------|-------------|---------|
| R1 | `[新增]` | `\"VIP\"` | `-` | `[0..100]` | `高` | 全场景高优先级 |
| R2 | `[新增]` | `\"普通\"` | `\"早晚高峰\"` | `(100..500]` | `中` | 高峰中等金额 |
| R3 | `[新增]` | `\"普通\"` | `\"非高峰\"` | `-` | `低` | 默认低优先级 |
| R4（兜底） | `[新增]` | `-` | `-` | `-` | `低` | **未命中默认行（必备）** |
```

## §8 BKM 内联 vs 抽出判据

| 场景 | 处置 |
|------|------|
| 公式**单次**在某决策表的结论列使用 | **内联**到结论列（不写 BKM） |
| 公式被 **≥ 2 个决策表**调用 | **抽出**为 `BKM-xxx`，决策表结论列改为 `BKM-xxx(参数1, 参数2)` |
| BKM 后续被**降至 1 处复用** | **主动回退**：内联回决策表，删除 BKM |

抽出过早 = 过度设计；不抽出导致重复 = 维护负担。**复用次数是唯一判据**。

---

## §9 主动回退（DMN 收敛后的反向纪律）

当业务规则收敛稳定（**一年不变 1 次**）、维度退化到 **≤ 2 个**时，应主动把决策表**回退为 Gherkin 或 L2 INV**，避免维护两套表达。

回退路径：

```
DMN 决策表  ─收敛稳定─▶  Gherkin Scenario 直写  ─或─▶  L2 INV 静态约束
```

回退动作在本 spec 标 `[已有·废弃]` DMN-xxx + 给出迁移路径与兼容期窗口（详见 [`./diff-and-migration.md`](./diff-and-migration.md) §4）。

---

## §10 校验规则（供 Stage 4 审计）

- 每张 DMN-xxx 决策表**显式标注命中策略**（不允许省略默认）；策略不得为 `F`
- 每张决策表**必含兜底行**（输入列全 `-`、输出列给默认值）
- 启用 DMN 时，对应 REQ 的 Gherkin Scenario 数量上限 = **命中策略分支数 + 1（兜底）+ 1（参数化）**；超过视为去重红线违例
- BKM 抽出条件 = 复用次数 ≥ 2；单次复用的 BKM 视为过度抽象
- `change_mode != greenfield` 时 `[已有·修改]` 决策表必附「规则迁移表」
- DMN 编号空间不与 `INV-x` / `AC-xxx-xx` 冲突
