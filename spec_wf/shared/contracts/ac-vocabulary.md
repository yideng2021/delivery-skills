# AC Vocabulary — INV / AC / DoD 三层口径权威定义

> **本文件是 v2 中 INV-x / AC-xxx-xx / DoD 三层口径的唯一权威定义**。
> 任何 skill / template / checklist 涉及这三个概念时**只能链接本文件**，不得复述定义。

---

## §1 三层口径速览

| 口径 | 中文名 | 层级 | 唯一来源 | 典型形态 |
|------|--------|------|----------|----------|
| `INV-x` | 业务不变量 | 数据级 AC | specs L2 业务实体与规则段 | 一句声明型规则 |
| `AC-xxx-xx` | 验收标准 | 功能级 AC | specs L3 行为规约 (EARS + Gherkin) | Given/When/Then 场景 |
| `DoD` | 完成定义 | 工程交付级 | specs L4 DoD 段 | checkbox 清单（**不新增验收点**，仅引用 INV/AC） |

> **核心约束**：AC 唯一来源 = `L2 INV` ∪ `L3 AC-xxx-xx`；`L1 不写 AC`；`L4 不新增验收点`。

---

## §2 INV-x — 数据级业务不变量

### §2.1 定义

声明业务实体 / 业务关系 / 业务状态层面**永真**的规则。与时间、操作、用户行为无关。

### §2.2 编号格式

`INV-{seq}`，`seq` 为 spec 文件内单调递增整数。例：`INV-1`, `INV-2`。

### §2.3 示例

```markdown
- **INV-1**：一个邮箱地址在系统中至多对应一个有效账户。
- **INV-2**：订单总金额 = 商品行金额之和 - 优惠金额（优惠不得超过商品总额）。
```

### §2.4 写作铁律

- 业务语言纯净 —— 不写"字段类型 / 1:N / 聚合根 / 数据库主键"等技术词
- 必须可验证（可由静态数据快照判定真伪）
- 不写"系统应当 ……"类操作型句式（那是 AC 的范畴）

## §3 AC-xxx-xx — 功能级验收标准

### §3.1 定义

声明系统在特定**触发条件 + 状态前提**下应表现的可观察行为。验收的唯一可执行依据。

### §3.2 编号格式

`AC-{req}-{seq}`：

- `req` = 该场景关联的需求标识（spec 内 EARS 句的关联键，通常对应 `REQ-xxx` 或 `AUTH-xx` 的尾段）
- `seq` = 同一 `req` 下的场景序号

例：`AC-signup-01`、`AC-signup-02`。

### §3.3 形态

每条 AC = 一段 Gherkin Scenario：

```gherkin
Scenario: AC-signup-01 — 访客用合法邮箱密码注册成功
  Given 访客未登录且邮箱 user@x.com 未注册
  When  提交合法邮箱与符合策略的密码
  Then  系统创建账户并跳转至主站
  And   邮箱进入"已激活"业务状态
```

### §3.4 写作铁律

- 每条 AC 必须可由 BDD 风格测试覆盖
- Then 子句应可观察、可断言；禁止"运行流畅"等模糊形容
- 不重复 INV-x 的内容；若行为本质是数据级规则，归 INV
- 不引入实现细节（HTTP code、字段名、SQL）—— 那是 design 与 dev skill 的事

---

## §4 DoD — 工程交付完成定义

### §4.1 定义

specs L4 段，是 spec 进入"已交付"状态的勾选清单。**不新增验收点**，仅以 checkbox 反向引用 INV-x 与 AC-xxx-xx，外加工程闭环（类型 / lint / 构建）与 L0 业务禁区守卫。

### §4.2 形态

```markdown
## L4 DoD
- [ ] INV-1（邮箱唯一性）数据快照测试通过
- [ ] INV-2（订单金额关系）静态校验通过
- [ ] AC-signup-01 BDD 用例通过
- [ ] AC-signup-02 BDD 用例通过
- [ ] 类型检查 / lint / 构建均通过
- [ ] L0 业务禁区未被触发（如未生成营销内容投放给未成年用户）
```

### §4.3 写作铁律

- **零新增验收**：每条 checkbox 必须能在 L2/L3/L0 中找到对应来源
- 不写"覆盖率 ≥ 80%"等工程指标（属下游 dev skill 的事）
- 不写具体测试命令（如 `npm test`）
- 工程闭环条目可统一为 1–2 条总结勾

## §5 与 specs 模板分层的映射

| specs 章节 | 对应 AC 口径 | 是否承载 AC |
|-----------|-------------|-------------|
| L0 业务上下文 | — | ❌（仅业务禁区参与 DoD 守卫）|
| L1 用户故事 | — | ❌（**显式不承担验收职责**）|
| L2 业务实体与规则 | `INV-x` | ✅ 数据级 AC 唯一来源 |
| L3 行为规约 | `AC-xxx-xx` | ✅ 功能级 AC 唯一来源 |
| L4 DoD | — | ❌（仅复选反向引用 INV/AC）|

---

## §6 与 frontmatter 字段的边界

- INV / AC / DoD 是 **specs 正文层** 的术语，**不进入** frontmatter 字段。
- frontmatter 中的 `related_req` 是 AUTH-ID 列表（项目级能力槽），与 INV/AC 编号空间正交，不可混用。
- 字段层定义见 [`frontmatter-schema.md`](./frontmatter-schema.md)。
