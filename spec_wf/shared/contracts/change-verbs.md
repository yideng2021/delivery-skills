# Change Verbs — 统一变更动词词表

> **本文件是 spec_wf 中 5 处近义枚举的共享语义源**。
> 任何 skill / schema / references 引用变更动词时,必须从本表 sub-select,不得自创。
> 词条一旦发布**不得改变语义**;新增词条须先扩本表。

---

## §1 核心词表(全集 9 个)

| 词 | 语义内核 | 典型场景 |
|----|---------|---------|
| `沿用` | 既有资产**原样保留**,本次不触碰其语义 | greenfield/extend 中既有 BC 不变;只读引用 |
| `扩展` | 既有资产**新增属性 / 场景 / 子能力**,**不改变**既有语义 | 新增字段;新增 Capability 子项 |
| `修改` | 既有资产**语义变化**或结构重组(行为 / 约束 / 数据形状改变) | INV 改、REQ 行为 Diff、字段语义变化 |
| `废弃` | 既有资产**本次起停用**(配套迁移路径 + 兼容期) | INV 退役、Capability 下线 |
| `替换` | 既有资产被**新资产代替**(语义不变或上位扩展) | 旧服务被新服务接管 |
| `新建` | 资产**首次引入**(此前不存在) | 新 BC / 新模块 / 新 Capability |
| `仅引用` | 既有资产作为**前置依赖 / 追溯锚**被引用,本次**不改、不扩** | reference_specs 中的既有 spec 锚 |
| `ACL隔离` | 跨 BC 间通过**反腐化层**接入(避免语义污染) | DDD Context Map 中的隔离边界 |
| `约束` | 资产作为**不可突破的边界**被引用(架构红线) | architecture_refs 中的总体架构约束文档 |

> **动词原则**:
> - 词由"对既有资产做什么"出发,不涉及"为什么做"或"做完是什么"
> - 多义词不允许;每个词必须语义内核唯一

## §2 五处枚举的 sub-select 映射

下表是 5 处枚举各自从本词表选了哪个子集。schema / references **不得**重新定义这些子集的语义,只能引用本表。

| 枚举出处 | 字段 / 上下文 | 合法子集 | 排除词及原因 |
|---------|-------------|---------|------------|
| **增量标注 5 项**(spec [`increment-annotation.md`](../../spec-writer-skill/references/increment-annotation.md) / `reused_modules.annotation`) | spec 业务规则 / design 既有模块复用 | `{[新增], [已有·仅引用], [已有·扩展], [已有·修改], [已有·废弃]}` | 不含 `沿用`(等同 `[已有·仅引用]`)/ `替换`(由"废弃旧+新增新"组合表达)/ `ACL隔离`(架构语义,非业务条目)/ `约束`(非动作) |
| **BC 关系 DDD 5 项**(`bc_relations.relation`) | design BC 与既有 BC 之关系 | `{沿用, 扩展, 新建, ACL隔离, 替换}` | 不含 `修改`(BC 一旦"修改"通常视为"替换"或"扩展");不含 `废弃`(BC 废弃应记 ADR 而非 bc_relations);不含 `仅引用` / `约束`(非 BC 间动作语义) |
| **architecture_refs.usage**(design `architecture_refs[].usage`) | design 引用的总体架构文档 | `{沿用, 扩展, 约束, 替换}` | 不含 `修改`(架构文档不由 design 修改)/ `废弃`(架构层不在 design 阶段废弃)/ `新建`(架构由专门 ADR/RFC 新建)/ `仅引用`(usage 强制有动作语义,纯只读用 `约束`)/ `ACL隔离`(不适用于"文档") |
| **proposal §0.1 关系列**([`existing-landscape.md`](../../proposal-writer-skill/references/existing-landscape.md) §3.1) | proposal 中既有 Capability / Spec / REQ / AUTH / ADR 与本次的关系 | `{沿用, 扩展, 修改, 废弃, 替换, 并存}` | 业务级粗粒度词表;`并存` 仅 proposal 阶段适用(粗略描述新旧两个均保留),不进入下游 spec/design 字段 |
| **proposal §0.2 触达方式**([`existing-landscape.md`](../../proposal-writer-skill/references/existing-landscape.md) §3.2) | proposal 中既有代码资产被本次如何触达 | `{读, 写, 改, 替换, 仅引用}` | 工程级粗粒度词表;`读/写/改` 与 `[已有·*]` 增量标注做语义衔接(不一一映射) |

## §3 多枚举之间的"动词同义关系"

下表说明同一动词在不同上下文中的对应写法(audit 时用于交叉校验):

| 语义 | 增量标注 | BC 关系 | architecture_refs | proposal §0.1 | proposal §0.2 |
|------|--------|--------|-----------------|--------------|--------------|
| 原样保留 | `[已有·仅引用]` | `沿用` | `沿用` | `沿用` | `仅引用` |
| 新增属性/场景 | `[已有·扩展]` | `扩展` | `扩展` | `扩展` | `读/写` |
| 语义变化 | `[已有·修改]` | (映射为"扩展"或"替换") | — | `修改` | `改` |
| 停用 | `[已有·废弃]` | (映射为"替换") | — | `废弃` | — |
| 新资产替代 | `[已有·废弃]+[新增]` | `替换` | `替换` | `替换` | `替换` |
| 首次引入 | `[新增]` | `新建` | — | — | — |
| 跨 BC 反腐化 | — | `ACL隔离` | — | — | — |
| 不可突破边界 | — | — | `约束` | — | — |

> 上表用于 audit 的"语义对齐检查":若 proposal §0.1 标 `修改` 但下游 spec 中无任何 `[已有·修改]` 标注,即视为追溯链断裂。

## §4 严禁事项 (Hard Bans)

- ❌ 自创词表外动词(如 `[改造]` / `[迁移]` / `[重命名]` / `合并` / `分裂` 等)
- ❌ 在 schema / references 中重新定义本表已有词的语义(只引用,不复述)
- ❌ 跨枚举混用("BC 关系"列出现 `[新增]` / 增量标注列出现 `ACL隔离` 等)
- ❌ 不经本表扩充直接在 SKILL.md 引入新枚举值

---

## §5 与 frontmatter-schema 的关系

- 本文件给**语义**,schema 给**机器约束**。
- schema `$defs` 中的 `changeVerb5` / `bcRelation` / `archUsage` 三个枚举均与本表 §2 的子集一一对应;若不一致以本表为准并同步更新 schema。
- audit 阶段 validator 不读本文件,但 audit 报告中可引用本表 §3 做"语义对齐检查"建议。
