# CDR Protocol — 注释驱动精炼协议

> **本文件是 v2 中 CDR(Comment-Driven Refinement,注释驱动精炼)的唯一权威定义**。
> 任何写手 skill / workflow 引用 CDR 时只能链接本文件,不得复述循环步骤、批注语法、退出条件。
> 本协议是横切机制,与具体阶段(proposal / specs / design / tasks)解耦。

---

## §1 协议定位

- **目的**:把"AI 生成 → 一次性通过"的失败模式,改写为"AI 生成 → 人类批注 → AI 修正"的迭代精炼,直至达成阶段验收标准。
- **核心命题**:Markdown 文档不仅是产物,更是**人机协作的共享可变状态**。批注是把人类判断注入文档的标准载体。
- **适用范围**:proposal-writer / spec-writer / design-writer / task-decomposer 4 个写手 skill 的每个产出阶段;不涉及 workflow 编排层。

---

## §2 触发条件

下列任一条件成立即必须执行 CDR:

1. 写手 skill 完成首版正文产出。
2. 用户在已产出文档中插入了批注(微观或宏观,见 §4)。
3. 验收标准(各 skill `references/checklist.md`)未全部满足。

> 写手 skill 的 SKILL.md 必须明确"产出后进入 CDR 循环",但**不重复定义**循环步骤。

---

## §3 循环步骤

```
┌──────────────┐
│ 1. AI 生成初稿 │
└──────┬───────┘
       ▼
┌──────────────┐
│ 2. 人类批注   │  ← Judgment Injection
└──────┬───────┘
       ▼
┌──────────────────────────────────────────────┐
│ 3. 安全锁指令:                                 │
│    "Read comments and update plan.            │
│     Don't implement yet."                     │
└──────┬───────────────────────────────────────┘
       ▼
┌──────────────┐
│ 4. AI 反向复述 │  ← 复述每条批注的理解
│   并修正文档   │  ← 修正后移除已解决批注
└──────┬───────┘
       ▼
   ◆ 5. 验收?
   Yes ──▶ 退出循环,文档 status: draft → reviewed
   No  ──▶ 回到第 2 步
```

> 步骤 1-4 构成"一轮"CDR;每阶段强制至少 1 轮,无上限。

---

## §4 批注语法

CDR 接受三种合法批注形态:

| 类型 | 语法 | 用途 | 示例 |
|------|------|------|------|
| 微观约束(行级) | `// 批注内容` 紧贴目标行尾,或独占下一行 | 行级精确修正、补充约束 | `// not optional` / `// 必须含分页` |
| 宏观重定向(段级) | `<!-- 批注内容 -->` 独占行,位于目标段落上方或内部 | 架构方向调整、整段重构指令 | `<!-- 拆分为两个独立服务,优先数据一致性 -->` |
| 安全锁指令 | `Read comments and update plan. Don't implement yet.` | 用户向 AI 发出"先读批注、暂不动手"的明确边界 | 整段独立成行 |

### §4.1 对话→批注自动转译(v3 新增)

> **背景**:实际场景中用户更倾向"对话式反馈"而非手写 inline 批注。强求 inline comment 反人类工程。
> **方案**:AI **必须**把用户聊天窗口的反馈主动落为文档批注,然后再走 CDR §5 反向复述。零判断丢失,完整审计链。

**触发条件**(任一即转译):
- 用户在聊天中表达对**当前阶段产物**的修改 / 补充 / 否决意图
- 用户口语化指代某节("第 3 条改成 X" / "把 §2 那个表去掉")

**转译形态**:

```markdown
<!-- comment-from-chat (@2026-05-17T14:23, turn-7): 用户原话「把第 3 条 INV 改成跨租户唯一性」 -->
```

**协议要点**:
1. **占位元数据**:必含 `(@时间戳, turn-N)` 便于 audit 追溯
2. **保留原话**:转译时**不**重新组织语言;原话用「」框选
3. **落位精确**:批注必须紧邻语义目标(目标条目 / 段落上方)
4. **不静默**:AI 在转译后必须显式回复"已记录为文档批注 @<位置>",让用户可校验
5. **可撤回**:用户回复"撤回上一条"即删除该批注(配套删除反向复述记录)

**转译边界**(不转译的情形):
- 用户的纯问题 / 澄清(无修改意图)
- 与当前阶段产物无关的闲谈
- 已在文档中存在的同义批注(避免重复)

> 详细举例与边界 case 见 [`../../{阶段}-writer-skill/references/how-to-write.md`](../../proposal-writer-skill/references/how-to-write.md) 的"CDR 分流"段落(各 skill 自治)。

---

## §5 反向复述要求

AI 在第 4 步修正前,必须对每条批注做**反向复述**,形态为 markdown 列表,内容回答 3 件事:

1. 理解的批注语义(用 AI 自己的话复述)。
2. 受影响的章节 / 字段 / 表项。
3. 拟采取的修正动作。

> 反向复述本身可临时写入文档"CDR Log"段或会话回复,不强制进入最终产物。但若用户要求保留可留作交付证据。

---

## §6 退出条件

同时满足下列条件方可退出 CDR 循环并将文档 `status` 由 `draft` 改写为 `reviewed`:

- [ ] 文档内已无未消化的批注(全部移除或被反向复述确认接受)
- [ ] 当前阶段 `references/checklist.md` 全部勾选 `[x]`
- [ ] 用户显式确认或停止追加批注超过约定次数

退出后,workflow 状态机据 `status: reviewed` 触发下一阶段转移(详见 [`../contracts/frontmatter-schema.md`](../contracts/frontmatter-schema.md) §3 字段写读流图)。

---

## §7 严禁事项 (Hard Bans)

- **One-shot**:首版直接通过、未经过任何批注循环视为违反 CDR。
- **跨阶段批注合并**:proposal 的批注不得拖到 specs 阶段处理;每阶段批注必须在其本阶段闭环。
- **AI 自行删除批注但未复述**:任何被移除的批注必须先在第 4 步反向复述中出现。
- **使用聊天窗口替代文档批注**:口头指令必须由 AI 主动落到文档批注上才进入 CDR(v3 起允许「对话→批注自动转译」,见 §4.1)。
- **复述式抄袭批注**:反向复述必须用 AI 自己的话,纯粹复制原批注文本不计入有效复述。

---

## §8 与各 skill 的关系(一句话)

| Skill | 在 CDR 中的角色 |
|-------|----------------|
| `proposal-writer-skill` | 在 proposal.md 上执行 CDR,直至战略共识达成 |
| `spec-writer-skill` | 在 specs/{capability}.md 上执行 CDR,直至 INV/AC 全部归零 |
| `design-writer-skill` | 在 design.md 上执行 CDR,直至架构契约清晰 |
| `task-decomposer-skill` | 在 tasks.md 上执行 CDR,直至工单边界与依赖关系清晰 |
| `spec-design-workflow` | **不参与 CDR**,仅监听各文件 `status` 字段以驱动阶段转移 |

> 各 skill SKILL.md 中关于"CDR 在本阶段如何展开"的细则**只能写一句**指向本文件,**禁止复述本文件内的循环步骤、批注语法、退出条件**。

---

## §9 校验规则(供 Stage 4 审计)

- 任何 SKILL.md / WORKFLOW.md / templates/* 出现"批注 / 注释循环 / 安全锁"等正文规则视为复述违例(链接除外)
- 出现旧术语如"复杂度守卫触发 CDR"等错绑定:CDR 与复杂度判定正交,审计失败
- workflow 状态机若直接读取批注文本而非 `status` 字段视为越界
