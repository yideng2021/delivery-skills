# AUTH-ID 命名规则

> 项目级需求 ID 的命名、分配、废弃规则。所有 RBK 用例必须遵循。

---

## 1. ID 结构

完整形式：`{CATEGORY}-{NUMBER}[{SUFFIX}]`

| 部分        | 规则                                                              | 示例           |
| ----------- | ----------------------------------------------------------------- | -------------- |
| `CATEGORY`  | 4 字母大写缩写（业务类别）                                        | `AUTH`, `CONT` |
| `-`         | 字面量分隔符                                                     | `-`            |
| `NUMBER`    | 同 Category 内严格递增的两位起步数字                              | `01`, `23`     |
| `SUFFIX`    | （可选）小写字母后缀，仅用于 D4 拆分场景                          | `a`, `b`       |

**示例**：
- `AUTH-01` — 标准形式
- `CONT-23` — Number 增长到两位以上仍保持原宽度
- `AUTH-10a`, `AUTH-10b` — D4 拆分场景（一条 AUTH 看似跨 Spec，拆为两条）

## 2. Category 命名规范

### 2.1 选取原则

- **业务语言优先**，避免技术词汇（`AUTH` 而非 `JWT`）
- **稳定性优先**，避免依赖具体功能（`CONT` 而非 `BLOG`）
- **可枚举性优先**，避免过于抽象（`PMT` 而非 `BIZ`）

### 2.2 推荐 Category 表

| Category | 全称              | 适用业务                      |
| -------- | ----------------- | ----------------------------- |
| `AUTH`   | Authentication    | 注册、登录、会话、验证        |
| `USER`   | User Profile      | 用户资料、头像、设置          |
| `CONT`   | Content           | 文章、评论、富文本            |
| `SOCL`   | Social            | 关注、点赞、私信              |
| `PMT`    | Payment           | 订单、支付、退款              |
| `NTFY`   | Notification      | 邮件、推送、站内信            |
| `ADMN`   | Admin             | 管理后台、权限                |
| `RPRT`   | Reporting         | 数据报表、统计                |
| `INTG`   | Integration       | 第三方对接                    |

> **首版策略**：优先复用上表；遇到新领域再扩展，并在本文档登记。

### 2.3 Category 重命名

**强烈不推荐重命名 Category**。如必须重命名：
1. 标记原 Category 为"已废弃"
2. 新 Category 从 `-01` 开始
3. 在 REQUIREMENTS.md 顶部添加 mapping 注释，便于历史检索

## 3. NUMBER 分配规则

### 3.1 标准分配

```
读取 REQUIREMENTS.md 中目标 Category 的所有 AUTH 条目（含撤销区）
  ↓
取 max(NUMBER) + 1
  ↓
若是新 Category：从 01 开始
```

**示例**：
```
现有：AUTH-01, AUTH-02, AUTH-05（AUTH-03/AUTH-04 已撤销，仍占位）
分配：max=5 → 新 ID 为 AUTH-06
```

### 3.2 NUMBER 永不复用

- **撤销不删行**：废弃需求迁移到 `## 撤销区` 段，原 ID 保留
- **不补空缺**：即使中间 ID 被撤销，新需求仍取 max+1，不填空

### 3.3 D4 拆分（特殊场景）

当一条 AUTH 在业务演化中需要拆分以满足 D4（一 AUTH 一 Spec）：

```
原：AUTH-10: 用户可通过多渠道登录
拆：AUTH-10a: 用户可用邮箱密码登录   → spec/email-login/
    AUTH-10b: 用户可用 OAuth 登录    → spec/oauth-login/
```

**规则**：
- 后缀使用小写字母 `a`, `b`, `c`...
- 拆分后原 ID `AUTH-10` 进入"已拆分"状态（同样在撤销区）
- 在撤销区行的"原因"列注明：`拆分为 AUTH-10a / AUTH-10b`

## 4. 撤销与归档

### 4.1 撤销流程

```
v1/v2 段中 AUTH-xx 被决定不做了
  ↓
将原行移动到 ## 撤销区 表格
  ↓
原行从 v1/v2 段删除（这是唯一允许的"删行"操作）
  ↓
不复用 ID
```

撤销区行格式：

```markdown
| AUTH-99 | 用户可用 SMS 登录 | 2026-05-06 | 改用邮箱验证替代 |
```

### 4.2 复活（已撤销的需求重新启用）

**禁止直接复活原 ID**。改为：
1. 从撤销区行的"原因"列追加"复活为 AUTH-NEW"
2. 分配新 ID 重新登记到 v1/v2 段
3. 历史保持可追溯

## 5. 校验规则（U6 audit 涉及）

| 规则           | 触发条件                                                          | 严重度 |
| -------------- | ----------------------------------------------------------------- | ------ |
| 格式不合规     | `auth-1`、`AUTH_01`、`AUTH-1` 等不符合 `{CAT4}-{NN[suffix]}` 格式 | 高     |
| Category 不一致 | 同一概念出现 `AUTH` 与 `Auth` 混用                                 | 中     |
| ID 重复        | 两条 AUTH 同号                                                    | 高     |
| ID 复用废弃 ID | 撤销区已有 AUTH-99，又分配给新需求                                | 高     |
| Number 跳号反推| `AUTH-99` 但缺乏 01–98 的足够基数                                  | 低（仅警告） |

详见 [`audit-checklist.md`](audit-checklist.md)。
