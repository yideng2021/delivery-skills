# 快照纪律（专治厚文档"冻结幻觉 / 随代码腐烂"）

> 厚导览天生有两个风险:① 把 AI 错误叙事冻成永久文件;② 随 commit 过期。
> 本文件定义对治这两者的强制纪律。违反即认为导览不可信。

---

## 一、回链(治幻觉)

每个断言——状态迁移、调用关系、跨域触点、约束——必须带 `文件:行号` 出处。
- 读者可一键核对源码;审阅者可抽查。
- AI 自检:"本句结论是否有代码出处?" 无出处不写。
- explore 输出被预算截断时,对具体符号补查,**不据截断片段推断**。

## 二、快照声明(治"被误当真理")

每份 `00-index.md` / `domain-*.md` 头部必须含声明块:

```
<!-- snapshot
generated_at: {{ISO8601}}
pin_commit: {{commit}}
generator: architecture-guide
scope: {{域名 | index}}
freshness: 教学用途。编程真理以活查询当前源码为准;本快照可能滞后于代码。
-->
```

正文首段重申一句:"本导览为 `{{commit}}` 时点的架构快照,供学习/交接;细节以源码为准。"

## 三、按域重生成(治腐烂)

代码演进后**不全量重写**,按域增量刷:

1. `gitnexus detect_changes`(或对比 `pin_commit`..HEAD 的 diff)→ 得到变更文件。
2. 把变更文件映射到业务域(包路径 → 域)。
3. **只重跑受影响域的 `domain-*.md`**;未受影响的域文件保持不动。
4. 跨域全景(00-index)若涉及变更域的主线,则相应更新该段。
5. 更新被刷新文件的 `pin_commit` / `generated_at`。

> 这正是"核心域拆成多文件"的回报:重生成粒度 = 单域,而非整份导览。

## 四、新鲜度自检(读时)

读者/agent 用前可比对 `pin_commit` 与当前 HEAD:
- 相等 → 新鲜。
- 落后 → 头部声明已警示;关键决策应回到活查询核对,或触发按域重生成。

---

## 反模式(禁止)

| 反模式 | 正确做法 |
|---|---|
| 把厚导览当"编程真理源",据其改代码不核源码 | 导览是教学快照;编程决策回活查询 verbatim 源码 |
| 无回链的"看起来很完整"的叙事 | 每断言带 文件:行号 |
| 代码变了就全量重写整份导览 | 按域 detect_changes,只刷受影响域 |
| 把厚内容写回 `01-architecture.md` | 01 保持薄(agent 用);厚内容置 `architecture/` |
| 据 explore 截断输出补全状态机 | 截断处补查后再写 |
