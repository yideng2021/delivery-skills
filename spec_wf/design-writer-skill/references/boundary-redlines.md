# boundary redlines(design 阶段边界红线)

> 本文件枚举 design.md 正文**禁止出现**的内容形态,是 design / spec / dev 三阶段边界纪律的执行指南。
> 验收清单见 [`./checklist.md`](./checklist.md) §3 边界红线。

---

## §1 红线总则

design 阶段产出的是**模块对外契约**与**领域建模骨架**,**不是**实现方案。任何"如何实现"的细节都属越界。

判定一段内容是否越界,只需问三个问题:

1. 这段内容是否能让 reviewer 在不查代码的情况下判断"业务对不对"?(是 → 留;否 → 删)
2. 这段内容是否一旦写错,会让 dev 阶段无法选择技术栈?(是 → 删,过早绑定)
3. 这段内容是否在 spec / proposal 阶段已经写过?(是 → 删,SSOT 违例;**正解 = 引用 + 架构层升级**,详见红线 7)

## §2 红线枚举(共 6 条)

### 红线 1:不出现 SQL / 表结构 / 字段类型

| 越界形态 | 纠正方向 |
|----------|----------|
| `CREATE TABLE users (id BIGINT, ...)` | 删除,归 dev 阶段 |
| `字段:user_id (int, NOT NULL)` | 改写为业务语言 "用户身份标识" |
| `主键 / 外键 / 索引` | 删除,归 dev 阶段 |

### 红线 2:不出现 HTTP / REST / 状态码

| 越界形态 | 纠正方向 |
|----------|----------|
| `POST /api/v1/users` | 删除,改为业务事件描述(例如 "用户注册请求") |
| `返回 401 / 403 / 500` | 删除,改为业务态描述(例如 "拒绝访问 / 服务异常") |
| REST 动词组合(GET/POST/PUT/DELETE) | 删除,归 dev 阶段 |

### 红线 3:不出现接口签名 / 代码片段 / 伪代码

| 越界形态 | 纠正方向 |
|----------|----------|
| `function authenticate(token: string): User` | 改写为模块对外契约的业务语言描述 |
| ```代码块``` | 删除整个代码块 |
| 伪代码 if-else 流程 | 改写为 §4 跨模块协作的业务事件流 |

### 红线 4:不出现框架名 / 版本号 / 库名

| 越界形态 | 纠正方向 |
|----------|----------|
| `使用 React 18 / Spring Boot 3.2` | 删除,归 dev 阶段 |
| `引入 axios / lodash / express` | 删除,归 dev 阶段 |
| 版本号 `v3.2.1` | 删除,归 dev 阶段 |

### 红线 5:不出现测试 / 部署 / 监控细节

| 越界形态 | 纠正方向 |
|----------|----------|
| `npm test / pytest / mvn test` | 删除,归 dev skill 与 spec L4 DoD |
| `覆盖率 ≥ 80%` | 删除,归 spec L4 DoD |
| 部署拓扑、监控指标具体名 | 删除,归 dev 阶段(可在 §5 ADR 提"需引入监控"但不写具体方案) |

### 红线 6:不复述既有契约文档

| 越界形态 | 纠正方向 |
|----------|----------|
| 复述 ac-vocabulary.md 的 AC 三层定义 | 删除,改为链接 |
| 复述 frontmatter-schema.md 的字段语义 | 删除,改为链接 |
| 复述 cdr-protocol.md 的 CDR 退出条件 | 删除,改为链接 |
| 复述 spec L0 业务禁区 | 删除,在 §6 越界声明用架构语言重述 |

### 红线 7:不复述上游阶段的"既有资产"原文(架构层 ≠ 业务层 ≠ 工程层)

design 关注**架构层**既有资产(BC / 模块 / 领域事件 / ADR),与 proposal §0(业务/工程级)+ spec L0.x(业务上下文级)**正交不重复**。**复述 = 越界**;**引用 + 升级 = 正解**。

| 越界形态 | 纠正方向 |
|----------|----------|
| 在 §1 复述 proposal §0.1 既有 Capability 行 | 删除;在 §2.1 BC 表中以"与既有 BC 关系"列承载(架构升级) |
| 在 §1 / §3.1 复述 proposal §0.2 既有代码资产表 | 删除;在 §3.1 模块清单的"既有模块路径"列以增量标注承载 |
| 在 §1 复述 spec L0.x 既有上下文衔接表 | 删除;design 关注 BC 关系(架构边界),非业务上下文(业务流程) |
| 在 §5 ADR 重抄既有 ADR 决策原文 | 改为"既有 ADR 引用"行 + 关系标注(`沿用` / `撤销` / `修订` / `取代`) |
| 在 §6.1 越界声明罗列 proposal §0 中所有"不动"条目 | 删除;§6.1 写**架构级**承诺,proposal §0 写**业务级**盘点 |

**判定口径**:**看视角,不看名字**——同一个"既有 user-service"出现在三阶段:proposal §0.2 关注"哪些代码被触达"(工程级),spec.impacted_modules 关注"业务规则影响哪些模块"(业务级),design.reused_modules + §3.1 关注"如何在架构上复用"(架构级)。**三视角应叠加而非复述**。

## §3 灰色地带的判定

下列内容**允许**出现,但需谨慎:

- **领域事件名称**(例如 "UserRegistered"):允许,这是业务概念,不是技术实现
- **限界上下文名称**(例如 "BC-Auth"):允许,这是建模产物
- **ADR 中提及"需引入消息队列 / 事件总线"**:允许,但**不指定**具体实现(不写 Kafka / RabbitMQ)
- **§3 模块对外契约的"输入 / 输出"**:允许写"业务概念"(例如 "用户身份凭证"),不允许写"字段类型"(例如 "string token")

### §3.1 架构层图表(允许 / 必要)——图 ≠ 字段

**重要澄清**:红线 1 禁的是"字段类型 / 表结构",**不是**所有图表。下列图表是架构层标准产物,**允许且必要**:

- **§2.2.2 聚合 ER 视图**(`erDiagram`):只画聚合根关系、聚合内核心实体,**禁字段类型 / 主外键 / 索引**。含字段类型的 ER 属于实现层,归 dev/database 阶段。
- **§3.2 模块依赖图**(`graph TB`):节点 = 模块名,箭头 = 同步依赖方向,**禁循环依赖**。不画类 / 函数 / 文件粒度。
- **§4 跨模块时序图**(`sequenceDiagram`):参与者 = §3.1 中存在的模块名,消息文案用业务语言,**禁具体方法签名 / HTTP 动词 / JSON 字段**。

判定口径:**看粒度,不看形式**——出现字段类型 / 方法签名 / HTTP 细节即越界,无论是否用图表承载。

## §4 越界发现后的处理

CDR 阶段 reviewer 发现越界,**必须**在批注中标注分流路径(见 [`./how-to-write.md`](./how-to-write.md) §CDR 反向分流):

- 红线 1-2-3 越界 → 转交 dev skill(工程闭环)
- 红线 4 越界 → 转交 dev skill(框架选型)
- 红线 5 越界 → 部分回 spec L4 DoD,部分转 dev skill
- 红线 6 越界 → 直接删除并改链接(在 design 内部消化)
- **红线 7 越界 → 直接删除,改为架构层升级**(BC 关系 / 模块增量标注 / ADR 引用;在 design 内部消化)
