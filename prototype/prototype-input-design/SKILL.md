# 原型输入设计

本 skill 专注于**把零散的产品/需求输入，转化为一份结构化的业务需求原型设计文档**。这是原型设计全流程的**前置阶段**，只关心"要设计什么"，不关心"怎么实现"。

**核心理念**：最小输入即启动，AI 主导整理，共创补全信息缺口，**只聚焦业务需求**，原型设计文档可追溯。

**输出物固定**：`docs/prds/<模块名>/input-design.md`（七要素 + 元信息 + 诊断决策）+ `iteration-log.md`（共创/确认过程记录）

**职责边界**
- ✅ 做：需求理解、信息整理、七要素完备性诊断、共创提问、推断标注、原型设计文档版本化维护
- ❌ 不做：任何代码相关事项——写 HTML/CSS/JS、设计视觉细节、决定组件样式、创建或读取 `_shared/` 目录、维护 `_index.md` 全局索引、读 `menu-data.js` 等代码文件、决定菜单如何挂载到代码中

## 何时使用本 skill

**适用场景**
- 用户提出新模块/新功能想做原型，但输入零散或模糊
- 用户想先把需求整理清楚再决定做不做原型
- 原型生成过程中发现需求有**结构性缺口**（新增页面 / 数据对象关系变化 / 操作流程变化 / 菜单挂载变化）需要回头补原型设计文档
- 已有原型但需要重新梳理原型设计文档以支撑后续迭代

**不适用场景（该用 `prototype-generation`）**
- 原型设计文档已齐全，直接要生成页面
- 只是调视觉细节（按钮色、间距、图标）
- 仅增加一个字段到现有页面（字段级追加）
- 反馈具体页面布局问题

**判别 tips**：如果用户语句里出现"生成"、"写页面"、"做个 xxx 页"、"改下样式"——优先 `prototype-generation`；出现"整理"、"需求"、"完善"、"先想清楚"——本 skill。

## 产物存储位置

本 skill **只动两个文件**，且都在模块目录下——**完全不碰 `_shared/`、`_index.md`、`prototype/` 这些实现层产物**（那些是 `prototype-generation` 的职责）：

```
docs/prds/
└── <模块名>/
    ├── input-design.md                # ⭐ 核心产物（A 全权）
    └── iteration-log.md               # 启动/共创/确认/变更（A 追加 [启动]/[共创]/[确认]/[强制生成]/[变更]）
```

**模块名规则**
- AI 基于用户输入推断（kebab-case，如 `task-management`）
- 首次使用时向用户确认（"准备将本次需求归入 `docs/prds/task-management/`，是否确认？"）
- 若 `docs/prds/<模块名>/` 已存在且有 `input-design.md`，读取其作为基线继续整理
- **不扫描 `docs/prds/` 下的兄弟模块或 `_shared/`**——不承担跨模块一致性的职责

## 核心流程（3 阶段）

```
阶段1 输入接收 → 阶段2 智能整理 → 阶段3 完备性诊断
                                        ↓
                                   信息不足时
                                 共创补全循环
                                        ↓
                              原型设计文档完备 → 明示切换到 prototype-generation
```

### 阶段 1：输入接收与识别

**做什么**
- 扫描用户输入，打类型标签：参考图 / 功能清单 / 菜单结构 / 业务流程 / 数据对象与关系 / 已有原型 / 口述想法 / 用户角色 / 用户故事 / 界面集成说明
- 将信息映射到"七要素"，标记已覆盖和缺失的维度 → 见 `references/seven-elements.md`
- 初步判断主基础方式与可能的叠加策略 → 见 `references/methods-and-strategies.md`

**不要做**：不跳过整理直接产原型设计文档；不向用户隐藏你的分类过程。

### 阶段 2：智能整理与结构化

**做什么**
- **持久化落盘**：
  1. 首次执行：向用户确认模块名（kebab-case），创建 `docs/prds/<模块名>/` 目录
  2. **无条件创建 `iteration-log.md` 并追加 `[启动]` 条目**（格式见 `templates/iteration-log.md`，加 `edited-by: A`）——记录用户指令原文、输入资料清单、本次处理路径，作为模块生命周期的基线审计。此步骤先于 input-design.md 的任何写入，且无论后续是否触发共创，本条目都必写
  3. 按七要素结构输出原型设计文档，写入 `docs/prds/<模块名>/input-design.md`（模板见 `templates/input-design.md`）
  4. 对未明示内容做合理推断，每条标注 `[AI推断-基于xxx]` 来源（来源限三类：用户输入 / 行业通用实践 / 项目已有上下文）
  5. 原型设计文档"元信息"节记录：模块名、业务名称、创建时间、当前状态（`整理中`）

**不要做**
- 不凭空捏造具体数值或不合理规则（如"默认保留 30 天"）
- 不使用用户未提及的业务术语
- 不跳过落盘直接进入下一阶段
- **不读任何代码层文件**（`_shared/`、`_index.md`、`menu-data.js`、兄弟模块的 `prototype/`）——这些是 B 的职责范围，A 读了反而会被实现细节污染

### 阶段 3：完备性诊断与共创补全

**做什么**
- 判断是否满足**最小输入条件**（仅七要素判定，不含任何实现细节）：
  - 模块结构（明确）+ 页面功能（明确）+ 数据对象与关系（基本明确）+ 操作流程（主流程明确）
  - 界面集成、业务规则、视觉参考**非必须**
- 输出诊断结论，**必须声明**推荐的"主基础方式 + 叠加策略"组合及理由（模板见 `templates/diagnosis.md`）
- 若不满足最小输入条件，选合适的共创补全策略引导用户（五种模板见 `templates/cocreation-patterns.md`）
  - 选择题式 / 预填确认式 / 示例引导式 / 定位辅助式 / 界面集成确认式
- 循环"补全 → 用户回答 → 更新原型设计文档 → 重诊断"直到达标
- **持久化落盘**：
  - 诊断结论（含推荐路径）**同步更新到 `input-design.md` 的"诊断决策"章节**
  - 在阶段 2 已创建的 `iteration-log.md` 中，**每轮共创问答作为 `[共创]` 条目追加**（格式见 `templates/iteration-log.md`，加 `edited-by: A`）
  - 用户回答引发的原型设计文档更新，**回写 `input-design.md`**
  - 原型设计文档状态更新为 `诊断完成`

**不要做**
- 不在输入不完整时强行推进（除非用户明确说"先按现在的生成看看"，此时在 iteration-log 中追加 `[强制生成]` 条目记录风险）
- 不向用户隐藏推荐的设计路径——必须让用户知道 AI 选了什么方式
- 不把共创问答只留在对话里——必须落盘到 iteration-log
- 不在诊断阶段讨论代码、视觉、组件、菜单实现——那是 B 的职责
- 不把"菜单怎么挂到系统"、"用哪些共享组件"作为诊断项——它们不是业务输入，是实现决策

## 交接协议（原型设计文档完备后）

原型设计文档到达 `诊断完成` 状态后，**必须**输出一段固定格式的引导，告知用户下一步切换 skill：

```
✅ 原型输入原型设计文档已就绪：docs/prds/<模块名>/input-design.md
   状态：诊断完成
   推荐设计路径：方式 X + 策略 Y

下一步：切换到 prototype-generation skill 进行原型生成。
  触发语示例："基于这份原型设计文档生成原型" / "按原型设计文档生成页面"

本 skill 会在以下情况被再次唤起：
  - 需求出现结构性变更（新增页面 / 改数据对象关系 / 改操作流程）
  - 生成过程中 B 识别到超出其权限的修改，转交回本 skill
```

## 与 prototype-generation 的协作契约

**A 对 `input-design.md` 的权限**：全权写入、重构、版本化维护

**B 对 `input-design.md` 的权限（本 skill 需了解）**：
- ✅ 允许：在"待确认事项"对应的字段级追加（如某字段最大长度、某默认值、某业务规则补充）
- ❌ 禁止：结构性变更（新增/删除页面、改数据对象关系、改操作流程、改业务规则结构）

**结构性变更路由规则**
- 若 B 在迭代中识别到需要结构性变更，会保持原型设计文档不动，并引导用户切回本 skill
- 本 skill 被再次唤起时，读取 `input-design.md` + 最近的 `iteration-log.md` 的 `[待转交]` 条目，定位待改节
- 更新后状态回到 `诊断完成`，再次明示用户切回 B

**状态机对齐**（`input-design.md` 元信息 `当前状态` 字段）
| 状态 | 主导方 | 本 skill 行为 |
|------|-------|-------------|
| `整理中` | A | 持续编辑原型设计文档 |
| `诊断完成` | A | 原型设计文档就绪，明示切到 B |
| `生成完成` / `迭代中` | B | 只读，除非收到结构性变更请求 |
| `已交付` | — | 只读 |

**代码层产物归 B 全权**（A 不感知不触碰）：`_shared/`、`_index.md`、`menu-data.js`、`prototype/`、菜单挂载位置、图标、接口路由、共享组件等。

## 关键原则

1. **最小输入原则**：总体信息具备即启动，不追求一次完美输入
2. **路径透明**：阶段 3 明示推荐的"方式 + 策略"，用户可否决调整
3. **推断可追溯**：所有 AI 推断必须标注 `[AI推断-基于xxx]`
4. **纯业务聚焦**：只关心七要素的业务需求，不讨论代码、视觉实现、菜单挂载、共享层——这些是 B 的职责
5. **职责边界**：本 skill 永远不写代码、不读代码文件、不维护实现层产物；有代码层问题引导到 B
6. **契约驱动**：`input-design.md` 是唯一的 A→B 交接契约，原型设计文档里没表达的业务信息，B 不会凭空生成
7. **产物持久化**：共创问答、诊断结论、推断来源统一落盘，不依赖对话记忆
