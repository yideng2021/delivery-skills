# 迭代记录 · 模板

> 阶段 2-5 持续追加。**保存路径**：`docs/prds/<模块名>/iteration-log.md`
> 时间线式记录，不覆盖历史。每条记录用 `[启动] / [共创] / [确认] / [变更] / [强制生成] / [待转交]` 标签区分类型，并标注 `edited-by: A/B` 标明由哪个 skill 写入。
> **模块首次受理时必写 `[启动]` 条目**，作为整个生命周期的输入基线。

---

## 使用说明

### 条目类型

| 标签 | 用途 | 写入方 | 触发阶段 |
|------|------|--------|---------|
| `[启动]` | 模块首次受理时的输入基线（指令原文、资料清单、处理路径） | A | 阶段 2 首次建模块，**无条件写入一次** |
| `[共创]` | 阶段 3 的共创补全问答 | A | 诊断发现缺失 → 提问用户 → 用户回答 → 回写原型设计文档 |
| `[确认]` | 待确认事项清单（生成过程中的 AI 假设） | B | 阶段 4 生成时产生 AI 推断/假设 |
| `[变更]` | 阶段 5 的迭代变更 | B（视觉/字段/业务规则/单页）/ A（结构性） | 用户提出修改 → 执行修改 → 记录影响 |
| `[强制生成]` | 用户在信息不完整时要求"先生成看看" | A 标记 → B 执行 | 阶段 3 诊断未通过但用户跳过补全 |
| `[待转交]` | B 识别到结构性变更，挂起并转交给 A | B | 阶段 5 反馈超出 B 权限 |

### 落盘时机

- **模块首次受理时，先建 `iteration-log.md` 并写入 `[启动]` 条目**（A，无条件，先于 input-design.md 的写入）
- 每一轮 AI 提问后立即追加 `[共创]` 条目，等用户回答后补全 `回答` 字段（A）
- 每次生成原型完成后，将本轮的 AI 假设集中写入 `[确认]` 条目（B）
- 每次用户反馈触发修改后，修改完成时追加 `[变更]` 条目
- B 识别到结构性变更时追加 `[待转交]` 条目，描述变更内容与转交理由（B）
- A 处理完 `[待转交]` 后在原型设计文档更新完成时追加 `[变更]` 条目回应

## 条目格式

### `[启动]` 条目（A 写入，每模块仅一次）

记录模块受理时的输入基线。所有后续变更、推断、共创都以本条目为追溯起点，**不做解读、不做关键词提取，只忠实保留用户指令原文与资料清单**。

```markdown
## [启动] 2026-04-21 10:30 — 需求计划
<!-- edited-by: A -->

**用户指令**（原文引用）：
> 根据 '需求计划.html' 这个原型设计，使用 /skill prototype-input-design
> 生成 '需求计划' 模块的原型设计文档，要求保证一比一完整还原需求计划原型设计的
> 界面布局和交互，包括待归档、已推送待接收、驳回修改、已归档 4 个过程的 tab
> 的不同列表字典，已经差异化的查询条件，必须根据界面代码完整、准确的整理清楚。
> 不能遗漏，也不能随意猜测。以上图片是界面的一些截图。

**输入资料**：
- 原型源文件：`需求计划.html`
- 参考截图：<N 张，路径或标识；若在对话中作为附件提供，此处记录附件编号或描述>
- 其他上下文：<无 / 相关文档路径 / 兄弟模块引用 等>

**处理路径**：直通整理（无共创） / 共创补全 / 强制生成（三选一）
**产出**：`input-design.md` · 状态 = `整理中` / `诊断完成`
```

**填写约束**：
- 用户指令**逐字引用**，不要总结或改写；过长时截取起止完整段落，中间可用 `……` 省略，但不得删改保留段
- 输入资料**必须列齐**所有被 AI 读取/参考的对象（文件路径、截图数量、口述要点来源等），缺一即为信息丢失
- 处理路径三选一必填，用于后续回溯"为什么没经过共创"等疑问

### `[共创]` 条目（A 写入）

```markdown
## [共创] 2026-04-19 14:30 — 操作流程补全
<!-- edited-by: A -->

**背景**：诊断发现"任务创建后的流转"不明确。
**补全策略**：选择题式
**AI 提问**：
> 任务创建完成后，下一步流向哪里？
>   A. 直接生效，返回列表页
>   B. 提交审核，审核通过后生效
>   C. 暂存为草稿

**用户回答**：A + C（允许直接生效，也允许暂存草稿）

**原型设计文档更新**：
- 操作流程节新增"主流程 / 草稿分支"两条
- 页面功能节"任务列表页"增加"草稿筛选 Tab"
- input-design.md 已回写
```

```markdown
|------|---------|------|------|------|

| 2 | 列表默认按"创建时间倒序" | [AI推断-通用实践] | ⬜ 待确认 | — |

```

### `[变更]` 条目（A 或 B 写入，按变更类型）

```markdown
## [变更] 2026-04-20 10:15 — 任务列表增加"导出"功能
<!-- edited-by: B -->

**变更类型**：功能增补（字段级，B 权限内）
**触发反馈**：用户提出"这个页面加个导出按钮"
**影响范围**：
- 新增 UI 元素：`pages/task-list.html` 表格右上角"导出"按钮
- 新增交互：点击弹窗选择导出字段/时间范围
- 新增接口：`_shared/api/routes.js` 追加 `EXPORT_TASKS`
- 新增 Mock：`<模块>/prototype/mock/export-task.js`

**原型设计文档更新（字段级追加）**：
- input-design.md "页面功能" 节 task-list 条目追加"批量导出"功能点（标注 `[迭代补充 by B]`）
- input-design.md 最后更新时间戳刷新

**决策理由**：导出功能不影响主流程，属于增量。沿用现有列表页，通过弹窗收集参数，保持风格一致。
```

### `[强制生成]` 条目（A 标记）

```markdown
## [强制生成] 2026-04-19 14:00 — 用户跳过补全，直接进入生成
<!-- edited-by: A -->

**缺失要素**：操作流程（主流程不完整）
**用户选择**：先生成看看
**风险提示**：
- 任务创建后的流转可能不符预期，需生成后确认
- 列表页的筛选维度可能遗漏重要筛选项

**AI 处理策略**：基于行业通用实践推断，相关推断在本文 `[确认]` 条目中集中列出。
```

### `[待转交]` 条目（B 写入，挂起转交给 A）

```markdown
## [待转交] 2026-04-21 11:20 — 任务审核流需要改为两级
<!-- edited-by: B -->

**反馈原话**：用户提出"任务审核流改为两级，组长 → 部门经理"
**超出 B 权限的原因**（原型设计文档结构性变更）：
- 改变原型设计文档"操作流程"节的结构（新增一个审核步骤）
- 可能需要新增"部门经理审核页"（涉及"模块结构"/"页面功能"节）
- 可能引入新数据对象"审核记录"（涉及"数据对象与关系"节）

**建议 A 操作**：
1. 读本条目定位变更点
2. 更新 input-design.md："操作流程"节、"模块结构"节、"页面功能"节、"数据对象与关系"节
3. 重新诊断 → 状态回到 `诊断完成`
4. 在本文追加 `[变更]` 条目（edited-by: A）关闭本转交

**B 当前处理**：挂起相关迭代；用户可先继续其他视觉/字段级/菜单挂载调整（后者 B 自己处理）。
```

## 维护建议

- **不删条目**：历史条目不删除，保留完整演进链路
- **最新在下方**：按时间正序追加，便于阅读发展脉络
- **关联引用**：涉及原型设计文档/代码变更时，在条目中引用文件路径
- **定期归档**：交付后可将本文标记为"归档版本"，后续 PRD 生成时作为一手输入
