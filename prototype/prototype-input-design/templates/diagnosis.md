# 完备性诊断输出 · 模板

> 阶段 3 必须输出。诊断结论必须包含**推荐的主基础方式 + 叠加策略**及其理由，让用户对 AI 的设计路径可感知、可调整。
>
> **落盘规则**：
> - 诊断结论（含推荐路径、最小输入条件达成情况、信息缺口处理）**同步写入** `docs/prds/<模块名>/input-design.md` 的"八、诊断决策"节
> - 若触发共创补全，每轮问答作为 `[共创]` 条目追加到 `docs/prds/<模块名>/iteration-log.md`
> - 用户回答引发的原型设计文档更新，**回写** `input-design.md`，并刷新"最后更新"时间戳
> - 原型设计文档状态：首次诊断通过后更新为"诊断完成"

## 情况一：满足最小输入条件 → 进入阶段 4

```
诊断结论：输入信息已具备生成条件。

✅ 模块结构：[明确 / 已识别 X 个模块，Y 个页面]
✅ 页面功能：[明确 / 已识别各页面核心功能和类型]
✅ 数据对象与关系：[基本明确 / 含 X 个业务对象，AI 推断补充已标注]
✅ 操作流程：[主流程明确 / 入口→操作→结果路径清晰]
ℹ️ 界面集成：[已指定 / 未指定，默认按常规新页面跳转处理]
ℹ️ 业务规则：[部分已知 / 可在迭代中补充]
ℹ️ 视觉参考：[已提供 XX / 无，AI 自主发挥]

→ 推荐设计路径：
   主基础方式：方式 X（名称） —— [选择理由，如"已有竞品截图作锚点"]
   叠加策略：[无 / 策略一：用户故事视角 —— 理由 / 策略二：多方案对比 —— 理由]
→ 如无异议，建议直接进入原型生成阶段。
```

## 情况二：不满足最小输入条件 → 进入共创补全

```
诊断结论：以下关键信息需要补充。

✅ 模块结构：明确
⚠️ 页面功能：部分模糊（XX 页的具体操作不清晰）
✅ 数据对象与关系：基本明确
❌ 操作流程：缺失（各页面间的跳转关系不明确）

→ 初步推荐路径（待信息补齐后最终确认）：
   主基础方式：方式 X（名称）
   叠加策略：[无 / 策略一 / 策略二 / 策略一+二]
   选择理由：[简要说明]

→ 需要补充以下信息（发起共创补全）：
   [使用 templates/cocreation-patterns.md 中的合适交互模板]
```

## 状态符号约定

| 符号 | 含义 |
|------|------|
| ✅ | 要素已明确，达标 |
| ⚠️ | 要素部分明确，需确认或补充 |
| ❌ | 要素缺失，必须补充 |
| ℹ️ | 要素非必须，可选 |

---

## 方式与策略推荐的模板句式

**主基础方式的表述**：
- `方式一（视觉参考驱动） —— 已有竞品截图/设计稿作锚点`
- `方式二（需求描述驱动） —— 功能需求清单明确，无视觉参考`
- `方式三（信息架构驱动） —— 已有完整菜单树，需全局搭建`
- `方式四（对话渐进式） —— 初始意图模糊，需渐进澄清`
- `方式五（存量迭代式） —— 基于现有原型做增量修改`

**叠加策略的表述**：
- `无 —— 单角色、方向明确`
- `策略一（用户故事视角） —— 存在多角色/注重体验验收`
- `策略二（多方案对比） —— 关键页面方向不确定/客户提案`
- `策略一 + 策略二 —— 多角色 + 关键决策并存`

## 触发"先生成看看"时的补充说明

当用户在信息不完整时主动要求"先生成看看"，诊断结论应追加：

```
⚠️ 用户选择在信息不完整时先生成，以下风险点请关注：
1. [缺失要素1]：可能导致 XX 不符合预期
2. [缺失要素2]：可能导致 YY 需要返工
生成时 AI 会对缺失内容做合理推断并在"待确认事项清单"中明确列出。
```
