# 原型设计七要素模型

> 判断输入完备性的核心框架。所有用户输入都归类到这七个维度。

## 七要素概览

| 要素 | 说明 | 最小输入要求 | 可 AI 推断 |
|------|------|-------------|-----------|
| 1. 模块结构 | 系统包含哪些模块/页面，菜单层级关系 | 至少明确当前要设计的模块包含的页面列表 | 可基于功能清单推断 |
| 2. 页面功能 | 每个页面的核心功能点 | 至少明确主要功能（如"列表查询""表单录入"） | 可基于页面类型推断常见功能 |
| 3. 数据对象与关系 | 核心业务实体、实体间关系（一对多/多对多等）、关键字段 | 至少明确核心业务对象及其主要属性 | 可基于业务领域推断通用对象和关系 |
| 4. 操作流程 | 用户的关键操作路径，页面间的跳转关系 | 至少明确主流程（入口 → 核心操作 → 结果） | 可基于功能关系推断基本流程 |
| 5. 界面集成 | 页面以什么位置、什么方式打开呈现（新页面/抽屉/弹窗/Tab 内嵌），是否有特殊集成要求（嵌入第三方、iframe、微前端） | 非必须（默认按常规页面跳转处理） | 可基于页面类型推断 |
| 6. 业务规则 | 校验规则、状态流转、权限控制等 | 非必须（可在迭代中补充） | 可基于行业通用实践推断 |
| 7. 视觉参考 | 参考图、风格偏好、已有设计 | 非必须（无则 AI 自主发挥） | 可基于项目已有页面保持一致 |

## 最小输入条件（生成门槛）

```
生成条件 = 模块结构(明确) + 页面功能(明确) + 数据对象与关系(基本明确) + 操作流程(主流程明确)
```

**前四项必须达标**，后三项非必须。

### 达标判据

| 判断维度 | 达标标准 | 未达标表现 |
|---------|---------|-----------|
| 模块结构 | 知道要设计哪些页面，页面间的菜单归属关系清晰 | 只有功能点列表，不知道怎么组织成页面和菜单 |
| 页面功能 | 每个页面的核心功能明确（至少知道这是什么类型的页面：列表/表单/详情/Dashboard） | 只有模糊描述，如"管理 XX" |
| 数据对象与关系 | 核心业务对象及其主要属性已知（可以有部分 AI 推断补充） | 完全不知道页面要展示和操作什么数据 |
| 操作流程 | 主流程的"入口 → 操作 → 结果"路径清晰 | 各功能孤立，不知道怎么串联 |

## AI 推断补充原则

| 原则 | 说明 |
|------|------|
| 基于输入推断 | 从用户已提供的信息中合理推导 |
| 基于行业通用实践 | 引用该业务领域的通用做法（如列表页通用功能：搜索/筛选/分页/导出） |
| 基于已有上下文 | 利用项目中已有的原型或代码保持一致 |
| 明确标注来源 | 每条推断内容标注 `[AI推断-基于xxx]` |
| 禁止凭空捏造 | 不编造具体业务数据、不假设不合理的业务规则 |

## 输入类型 → 要素映射

帮助阶段 1 快速判断哪些要素已被覆盖：

| 用户输入类型 | 主要覆盖要素 |
|-------------|-------------|
| 竞品/参考截图 | 视觉参考；部分覆盖模块结构、页面功能 |
| 功能清单 | 页面功能；部分覆盖模块结构 |
| 菜单/模块结构（树） | 模块结构；部分覆盖操作流程 |
| 业务流程描述 | 操作流程；部分覆盖业务规则 |
| 数据对象/ER 图/字段列表 | 数据对象与关系 |
| 界面集成说明 | 界面集成 |
| 已有原型/代码 | 视觉参考、模块结构、页面功能（多要素覆盖） |
| 口述想法 | 覆盖度低，通常需要共创补全 |
| 用户角色/用户故事 | 操作流程、业务规则（角色权限） |
