# 原型生成

本 skill 专注于**把结构化的 `input-design.md` 翻译成符合规范的、可直接演示的 Web 高保真 HTML 原型**。这是原型设计全流程的**生成与迭代阶段**。

**核心理念**：原型设计文档驱动生成；项目内模板模块驱动风格基线（`_template-<类型>/` 是真实可运行模块，与业务模块同级同形）；索引（`_templates.md`）是本 skill 唯一入口；SSOT 防止风格漂移；`file://` 原生可用；迭代有边界。

## 输入契约

| 输入 | 位置 | 来源 | 本 skill 权限 |
|------|-----|------|--------------|
| 业务输入 | `docs/prds/<模块>/input-design.md` | `prototype-input-design` 产出（状态 = `诊断完成`+） | 只读；允许字段级追加 |
| 模板索引 | `docs/prds/_templates.md` | 由独立的模板策展 skill（上层智能体调度）维护 | **只读** |
| 模板模块 | `docs/prds/_template-<类型>/` | 同上，模板的实体代码 | **只读**（作为生成参考） |
| 共享层 | `docs/prds/_shared/` | 由模板策展 skill 初始化；本 skill 复用 | 字段级追加（菜单项 / 路由常量）；不重写结构 |
| 全局索引 | `docs/prds/_index.md` | 首个模块生成时由上游创建 | 追加本模块注册行 + 刷新菜单快照 |

## 输出物

| 产物 | 位置 | 职责 |
|------|-----|------|
| 业务模块原型 | `docs/prds/<模块>/prototype/` | pages / css/pages / js/{services,pages} / mock |
| 模块迭代记录 | `docs/prds/<模块>/iteration-log.md` | 本 skill 写 `[确认]` / `[变更]` / `[待转交]` |
| 全局索引追加 | `docs/prds/_index.md` | 在"模块注册表"追加本模块行；"菜单结构快照"同步 |
| 菜单注册追加 | `docs/prds/_shared/reference/menu-data.js` | 追加本模块菜单项（不改已有项结构） |

## 职责边界

**✅ 做**
- 读 `_templates.md` 找匹配模板
- 基于选中的模板模块 `_template-<类型>/prototype/` 的页面模式生成业务模块原型
- 决定本业务模块的菜单挂载位置（有歧义时向用户确认）
- 追加菜单项到 `_shared/reference/menu-data.js`
- 追加接口常量到 `_shared/api/routes.js`
- 维护 `_index.md`（模块注册 + 菜单快照）
- 处理视觉 / 字段级 / 业务规则级 / 单页迭代

**❌ 不做**
- 不创建模板；不修改 `_templates.md`；不修改 `_template-<类型>/` 内容
- 不初始化 `_shared/`（若 `_shared/` 缺失说明模板未就绪，挂起并输出引导文本）
- 不做人机共创模板初始化（属于模板策展 skill 的职责，本 skill 全程无感知）
- 不重写 `input-design.md` 的结构性内容（新增/删除页面、改数据对象关系、改操作流程、改业务规则结构）
- 不调用 fetch / 动态 import / 本地 HTTP server

## 何时使用本 skill

**适用场景**
- `input-design.md` 已就绪（状态 = `诊断完成`+），要首次生成业务模块原型
- 已有业务模块原型，用户反馈"改按钮色"、"加导出按钮"、"调布局"等迭代需求
- 已有业务模块原型，用户要"换种方式展示同样的数据"（策略二多方案）

**不适用场景**
| 情况 | 正确路径 |
|------|---------|
| `input-design.md` 还未整理 / 诊断未通过 | 切换到 `prototype-input-design` |
| `_templates.md` 不存在或无匹配模板 | **输出引导文本挂起**，由上层智能体调度模板策展 skill |
| `_shared/` 不存在（模板未就绪） | 同上 |
| 用户说"加一个审核流程"、"新增统计报表页"、"改数据模型" | 切换到 `prototype-input-design`（属原型设计文档结构性变更） |
| 用户说"新增一种模板风格" / "换一套视觉风格模板" | 切换到模板策展 skill |

**判别 tips**
- 凡是"加/删**页面**"、"改**操作流程**"、"改**数据对象关系**"、"改**业务规则结构**" → 结构性变更，转回 A（prototype-input-design）
- 凡是"新增模板"、"换模板" → 模板策展 skill
- 其他（视觉 / 字段 / 单业务规则补充 / 单页面内增删按钮 / 菜单挂载调整） → 本 skill 自己处理

**关键约定**
- `_template-<类型>/` 是**真实模块**，与业务模块同级同形；模板界面都统一会在项目菜单"📚 模板参考"一级分组下，以各页面类型为二级或三级项。
- 本 skill 对模板模块**只读**，把它作为"可视化 + 可运行的参考文档"，而不是拷贝源
- `_templates.md` 是本 skill 的索引入口——它不存在 / 为空 / 无匹配，都触发"挂起 + 引导文本"

## 核心流程（2 阶段）

```
阶段 4 原型生成
   ↓ 模板索引就绪 → 匹配成功 → 生成
   ↓ 模板索引未就绪 / 无匹配 → 输出引导文本 → 挂起
阶段 5 迭代优化
   ↓ 结构性变更 → 转交回 prototype-input-design
   ↓ 模板相关变更 → 转交给模板策展 skill
   ↓ 其他 → 本 skill 直接处理
```

### 阶段 4：原型生成

**启动动作（先读再写，顺序不可省）**

1. 读 `docs/prds/<模块>/input-design.md`
   - 若状态 = `整理中` → 停止并提示用户切换到 `prototype-input-design`
   - 若状态 = `诊断完成` 或更高 → 继续

2. 读 `docs/prds/_templates.md`
   - 若文件不存在 → **输出"模板未就绪"挂起文本**（见下）
   - 若文件存在但索引表为空 → 同上
   - 若文件存在且有条目 → 进入匹配（详见 `references/template-lookup.md`）

3. 读 `docs/prds/_shared/reference/menu-data.js`（了解现有菜单）
   - 若 `_shared/` 缺失但 `_templates.md` 有内容 → 数据不一致，**挂起**并提示用户：模板索引与共享层不一致，需要模板策展 skill 校正

4. 读 `docs/prds/_index.md`（了解兄弟模块分布），不存在则在本次生成结束时补创建

**模板匹配**

按 `references/template-lookup.md` 执行匹配信号对齐 → 给出高/中/低评估 + 理由 → 向用户确认。

三种分支：

| 分支 | 判定 | 动作 |
|------|-----|------|
| 有匹配 | 用户采纳某个候选模板 | 进入生成 |
| 无匹配 | 索引有内容但所有候选匹配度低、或用户否决 | **输出"无匹配"挂起文本**并结束本轮；由上层智能体决定是否调度模板策展 skill |
| 未就绪 | `_templates.md` 缺失或为空 | **输出"模板未就绪"挂起文本**并结束本轮；同上 |

**挂起文本模板（直接输出给用户，不要加装饰）**

_模板未就绪_：
```
当前项目的模板索引 `docs/prds/_templates.md` 尚未就绪（不存在或为空）。
本 skill 依赖此索引进行风格基线匹配，无法在未就绪状态下生成原型。

建议：切换到模板策展 skill 完成项目模板库初始化。完成后回到本 skill 继续生成 <模块名> 的原型。

提示：模板库初始化是一次性工作。首次完成后，后续业务模块可直接匹配复用。
```

_无匹配_：
```
已读取模板索引 `docs/prds/_templates.md`，当前候选均与本模块的视觉参考 / 页面类型分布不契合：
  - <模板 id 1>：匹配度 低（理由：...）
  - <模板 id 2>：匹配度 低（理由：...）

本 skill 不创建新模板。建议：切换到模板策展 skill 新增一个契合 <模块名> 的模板（形态：... / 页面类型：...）。完成后回到本 skill。
```

**决定菜单挂载位置**

基于 `input-design.md` "模块结构"节的业务归属描述 + 已有菜单结构（读 `_shared/reference/menu-data.js`），决定：
- 一级分组（优先复用已有分组；需新增时向用户确认分组名）
- 菜单项名（建议与业务名一致）
- 图标（基于业务属性选 emoji 或图标库名；有歧义时向用户快速确认）
- 顺序（同级追加到末尾）

**生成顺序**

1. **读选中模板模块作参考**：`docs/prds/_template-<类型>/prototype/pages/`，了解列表/详情/表单页的页面模式、样式组织、交互约定；不 copy 代码，只借鉴模式
2. **菜单注册（硬性前置）**：把本模块菜单项 push 到 `_shared/reference/menu-data.js`；未注册不得生成页面
3. **模块骨架**：创建 `<模块>/prototype/` 下 `pages/` / `css/pages/` / `js/{services,pages}/` / `mock/`
4. **核心页面优先**：使用频率最高或业务最关键的页面（通常是列表页或仪表盘）
5. **关联页面 → 辅助页面**（详情 / 表单 / 设置等）
6. **Mock 数据填充**（`window.XXX_MOCK` 常量，业务合理，覆盖正常/极值/异常三态）
7. **全局索引更新**：在 `_index.md` 模块注册表追加本模块条目（状态 + 原型入口 + 菜单挂载 + 参考模板 ID）；刷新"菜单结构快照"节

**单页面生成顺序**
1. 引用共享资源（`<link>` tokens.css/common.css + `<script src>` menu-data/header/sidebar/common/routes），不 copy-paste
2. 页面骨架（顶部 / 内容区 / 底部划分）
3. 核心区域（表格 / 表单 / 卡片 / 图表）
4. 操作交互（按钮 / 搜索 / 筛选 / 分页）
5. 状态处理（空态 / 加载骨架屏 / 错误态 / 操作 Toast）
6. Mock 数据填充
7. 规范三重自检（界面 + 工程 + file:// 兼容，见 `references/engineering-standards.md` 第十二节）

**持久化落盘**
- 生成完成后，在 `<模块>/iteration-log.md` 追加 `[确认]` 条目（`edited-by: B`），列出本轮所有 AI 假设（含参考模板 ID、菜单挂载决策、图标选择等）
- 更新 `<模块>/input-design.md` 元信息：`当前状态 = 生成完成`，刷新"最后更新"时间戳（**仅改元信息**）
- 首次全局：补创建 `_index.md` 并登记本模块；非首次：追加模块注册表行 + 刷新菜单快照
- 若追加了全局接口常量：在 `_shared/iteration-log.md` 追加 `[变更]` 条目

**不要做**
- 不生成前不读原型设计文档——凭对话记忆生成会失焦
- 不跳过模板索引——`_templates.md` 缺失时**必须**挂起，不能凭手感直接生成
- 不修改 `_templates.md` / `_template-<类型>/`（只读参考）
- 不只设计"理想状态"而忽略空态 / 错误态 / 加载态
- 不在 HTML 内写海量行内样式或业务逻辑
- 不页面间 copy-paste 公共组件（header/sidebar）
- 不在模块内自建 `routes.js` / `header.js` / `tokens.css`——共享资源必须走 `_shared/`
- 不先生成页面后补菜单——**菜单注册是硬性前置**
- 不使用 `fetch()` / 动态 `import` 加载本地文件（违反 file:// 兼容）
- 不用无意义 Mock 数据（`test1`、`111`）

### 阶段 5：迭代优化

**反馈分类与路由**

| 反馈类型 | 示例 | 由谁处理 | 是否改原型设计文档 |
|---------|------|---------|-----------|
| 视觉 / 代码微调 | "按钮颜色深一点"、"间距再大点" | B | 否 |
| 菜单挂载调整 | "挪到设置分组"、"换一个图标" | B（改 `_shared/reference/menu-data.js` + `_index.md`）| 否 |
| 字段级补充 | "任务加个优先级字段" | B | 是（字段级追加）|
| 业务规则补充 | "延期超 3 天自动预警" | B | 是（字段级追加）|
| 整页重做（同需求） | "这个布局不行，换种方式" | B（叠加策略二）| 否 |
| 跨模块 token 微调 | "主色再深一点" | B（改 `_shared/css/tokens.css`，列出受影响模块）| 否 |
| **新增模板 / 换模板风格** | "来一套现代风模板"、"换一套视觉风格" | **转交模板策展 skill** | 否 |
| **原型设计文档结构性变更** | "新增审核页"、"任务改两级审核"、"改数据对象关系"、"主流程走向变了" | **转交回 A（prototype-input-design）** | 是（由 A 重写相应节）|

**原型设计文档结构性变更判别（启发式）**
> 如果改动要**改写原型设计文档七要素中某一节的结构**（新增/删除/关系变化），就转给 A；否则 B 自己处理。

具体对照：
- 新增/删除**页面** → A（动"模块结构"/"页面功能"）
- 改**数据对象**（新增对象 / 删除字段 / 改对象间关系）→ A（动"数据对象与关系"）
- 改**操作流程**（新增步骤 / 改主流程走向 / 增加分支流程）→ A（动"操作流程"）
- 改**业务规则结构**（推翻已定规则 / 引入成体系的新规则集）→ A（动"业务规则"）
- 改**菜单挂载**（换分组 / 换顺序 / 换图标） → B（这是实现决策，不动原型设计文档）
- 其他（视觉 / 单字段追加 / 单条业务规则补充 / 页面内控件增删）→ B

**转交到 A 的动作**：输出如下引导并在 iteration-log 追加 `[待转交]` 条目（`edited-by: B`）：

```
此调整涉及 [具体结构，如"操作流程新增审核步骤"]，超出本 skill 职责。

建议切换到 prototype-input-design skill 更新原型设计文档，理由：
  - 此变更会改变原型设计文档的"操作流程"节 / 增加新页面登记
  - 新页面的界面集成、业务规则需要重新诊断

待原型设计文档更新完成（状态回到 `诊断完成`）后，切回本 skill 继续生成 / 迭代。
```

**转交到模板策展 skill 的动作**：同样输出引导 + 追加 `[待转交]` 条目：

```
此调整涉及模板风格或模板库结构，超出本 skill 职责。

建议切换到模板策展 skill，理由：
  - <具体理由，如"需要新增数据大屏类型的模板"或"现有商务风模板整体调整">

完成后回到本 skill 继续当前模块的生成 / 迭代。
```

**B 处理的迭代**（每次都必须做的）
1. 先读三份文件：`<模块>/input-design.md` / `<模块>/iteration-log.md` / `_shared/iteration-log.md`（全部从文件读，不靠对话记忆）
2. 按反馈类型改对应位置：
   - 模块内视觉 / 交互 → 改 `<模块>/prototype/` 对应文件
   - 跨模块视觉 / token → 改 `_shared/css/tokens.css` 或 `_shared/css/common.css`；**必须列出受影响模块清单并验证不破坏**
   - 菜单结构（挂载 / 顺序 / 图标 / 分组） → 改 `_shared/reference/menu-data.js` + 同步 `_index.md` 菜单快照
3. 落盘：
   - 追加 `<模块>/iteration-log.md` 的 `[变更]` 条目（`edited-by: B`）
   - 共享层变更追加 `_shared/iteration-log.md` 的 `[变更]` 条目
   - 字段级原型设计文档追加：在 `input-design.md` 对应节追加（**只追加，不重写结构**），刷新时间戳
   - 状态变更：`生成完成` → `迭代中`；收敛后 → `已交付`，同步 `_index.md`

## file:// 兼容（硬性前置约束）

本 skill 产出必须支持直接双击打开运行，不起任何本地 server：

**禁止使用**
- ❌ `fetch()` / `XMLHttpRequest` / ES Module 动态 `import`
- ❌ 硬编码绝对路径（`/docs/prds/...`）

**必须使用**
- ✅ `<script src="...">` 同步加载 JS
- ✅ `<link rel="stylesheet" href="...">` 加载 CSS
- ✅ 公共组件以 JS 字符串模板形式存在（`HEADER_HTML`），由渲染函数挂到 DOM
- ✅ Mock 数据作为 `window.XXX_MOCK` 常量，通过 `<script src>` 加载

详细约束见 `references/engineering-standards.md` 第二节与第十二节自检清单。

## 关键原则

1. **原型设计文档驱动**：每次生成 / 迭代开始必先读 `input-design.md` 与相关 log；不靠对话记忆
2. **模板索引驱动**：首次生成前必先读 `_templates.md`；索引未就绪 / 无匹配即挂起，不自建模板、不凭手感
3. **规范先行**：每次生成自动注入界面规范 + 工程规范 + 模板查找规程
4. **全局 SSOT**：跨模块公共资源（header / sidebar / token / 菜单 / 路由）只在 `_shared/` 定义一次
5. **file:// 原生可用**：禁 fetch / 禁动态 import / 禁本地 HTTP
6. **菜单注册前置**：新模块生成前必须先在 `_shared/reference/menu-data.js` 登记
7. **职责边界清晰**：结构性变更转 A；模板相关转模板策展 skill；其他 B 自理
8. **变更留痕**：每次改动有对应 iteration-log 条目
9. **产物持久化**：过程日志、代码、状态统一落盘到 `docs/prds/`；不依赖对话记忆
